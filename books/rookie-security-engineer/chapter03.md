## 典型的な1日のタイムライン

セキュリティエンジニアの「1日」は、職種で大きく変わります。代表的な3パターンを見てみます。

### SOCアナリスト(シフト制の場合)

```
09:00 シフト引き継ぎ・前任からの申し送り確認
09:30 SIEMのアラートをトリアージ(優先度仕分け)
11:00 検知ルールのチューニング
13:00 昼休憩
14:00 顧客向けインシデントレポート作成
16:00 脅威インテリジェンスの読み込み
18:00 次シフトへ引き継ぎ
```

### Webアプリ脆弱性診断員

```
10:00 朝会・本日対象システムの確認
10:30 Burp Suiteで対象サイトを探索
12:00 発見した挙動を手動で再現・深掘り
13:00 昼休憩
14:00 リクエスト改ざんで脆弱性を検証
17:00 報告書の所見を書く
19:00 退勤
```

### セキュリティコンサル

```
09:00 メール処理・顧客対応
10:00 ISMS構築支援の打ち合わせ
12:00 昼休憩
13:00 リスクアセスメント結果のドラフト
16:00 提案書レビュー
18:00 顧客向け資料作成
20:00 退勤
```

毎日こんなにきれいに流れるわけではありませんが、職種で **時間の使い方が驚くほど違う** ことは伝わるかと思います。

## 緊急対応のリアル

平時の業務とは別に、セキュリティエンジニアには **緊急対応(インシデント対応)** があります。

ランサムウェアに感染した、機密情報が漏洩した、不審なログインが検知された――そうした事態が起きると、深夜・休日関係なく招集されることもあります。事業会社の社内セキュリティやMSSベンダーでは、オンコール(待機当番)制度があるのが一般的です。

ただし、ハリウッド映画のように毎日が緊急対応というわけではありません。多くの会社では年に数回、深刻なものは数年に一度。それでも備えと訓練は欠かせません。

## 攻撃者目線と守る側目線の往復

セキュリティエンジニアに最も求められるマインドセットは、**「攻撃者ならどう考えるか」と「守る側としてどう対処するか」を行き来できる思考** です。

例えば、新しいログイン機能を設計するとき、開発者は「ユーザーが使いやすいか」を考えます。セキュリティエンジニアはそこに「攻撃者ならどこを突くか」「突かれたとき何が起きるか」「起きたあとどう検知・復旧するか」のレイヤーを重ねます。

この往復思考は一朝一夕には身につきません。CTFや脆弱性診断のハンズオン、過去のインシデント事例の学習を通じて、少しずつ鍛えていきます。

## 学び続けることが前提の職業

最後に、心構えとして一つ。

**セキュリティは「学び続けることが仕事」の職業** です。攻撃手法は毎月のように更新され、新しいクラウドサービス、新しい暗号アルゴリズム、新しい法令が次々と登場します。

「資格を取れば一段落」という業界ではありません。これは大変さでもあり、面白さでもあります。日々ニュースを読み、コードを書き、論文に目を通す――そういう生活が楽しいと思える方には、間違いなく向いている職業です。

ここまでで業界の全体像はつかめたはずです。次のPart 2からは、いよいよ **攻撃と防御の基礎知識** に入っていきます。
