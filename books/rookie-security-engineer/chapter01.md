## 「セキュリティエンジニア」という言葉の曖昧さ

「セキュリティエンジニアになりたい」と話したとき、相手によって思い浮かべる仕事像はバラバラです。

ある人はハッカー映画のように黒い画面でコマンドを叩く姿を、別の人は深夜にアラートを監視する人を、また別の人は分厚い規程文書を書くコンサルタントを思い浮かべるでしょう。

実はそのすべてが「セキュリティエンジニア」です。この言葉は **役割の総称** であり、具体的な仕事内容は専門領域によって大きく異なります。だからこそ、最初に職種マップを頭に入れることが重要です。

## 職種マップ全体図

セキュリティエンジニアの仕事は、おおまかに5つに分類できます。

| 分類 | 仕事の中心 | 代表的なロール |
|------|-----------|---------------|
| ブルーチーム | 守る側 | SOCアナリスト / CSIRT / 検知エンジニア |
| レッドチーム | 攻める側(合法的に) | 脆弱性診断員 / ペネトレーションテスター |
| プロダクトセキュリティ | 自社サービスを安全に作る | 社内セキュリティエンジニア / DevSecOps |
| ガバナンス・コンプライアンス (GRC) | ルールと監査 | セキュリティコンサル / 監査人 |
| 研究・開発 | 新しい脅威と対策の研究 | マルウェアアナリスト / セキュリティリサーチャー |

これらは互いに重なる部分もあります。例えば「Webアプリ脆弱性診断員」はレッドチームに分類されますが、診断結果をもとに開発者へ助言する点ではプロダクトセキュリティとも接します。

## 隣接職種との違い

セキュリティエンジニアと混同されやすい職種に、**SRE(サイト信頼性エンジニア)**、**インフラエンジニア**、**ネットワークエンジニア** があります。

ざっくり言えば、

- インフラ/ネットワークエンジニアは「**動かす**」のが仕事
- SREは「**安定して動かし続ける**」のが仕事
- セキュリティエンジニアは「**正しい人だけが、正しい使い方で動かせる状態を保つ**」のが仕事

です。重なる領域は多く、実際にインフラエンジニアからセキュリティエンジニアへ転身するキャリアパスもよくあります。

## どの職種から入るのが現実的か

未経験から入る場合、最も門戸が広いのは **SOCアナリスト** と **Webアプリ脆弱性診断員** です。

SOCはアラートを監視・分析する仕事で、シフト勤務のチーム体制が多いため新人を育てる仕組みが整っています。脆弱性診断員は手を動かして「やった結果」が見えやすい仕事で、CTF(Capture The Flag、後述)で腕を磨いた学生が就職しやすい職種でもあります。

逆に、コンサルや監査の仕事は業務経験(IT全般や事業会社経験)が前提になることが多く、新卒で直接入るのは比較的少数派です。

第2章では、これらの職種がどんな **会社** にあるのか、業界構造を見ていきます。
