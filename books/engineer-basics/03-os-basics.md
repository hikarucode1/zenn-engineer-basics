## この章のゴール

第1章でハードウェアの登場人物、第2章でデータの正体を見ました。
ここで疑問が出てきます — **「あなたが書いた Python のプログラムは、どうやって CPU の上で動いているの?」**

直接 CPU を叩いているわけではありません。間に **OS (Operating System / オペレーティングシステム)** という巨大な仲介者がいます。
Windows、macOS、Linux、iOS、Android — これらはすべて OS です。この章では「OS が何をしてくれているか」を俯瞰します。

## 3-1. OS は何を解決してくれるのか

もし OS がなかったら、あなたは次の全部を自分で書く必要があります。

- CPU を直接操る命令
- メモリのどこに何を置くかの管理
- ストレージの管理 (どのセクタにファイルを書くか)
- キーボード入力やネットワークパケットの受信処理
- 他のプログラムとの共存 (どっちが先に CPU を使うか)

これは現実的じゃありません。OS は **「ハードウェアの面倒ごとを引き受けて、アプリには綺麗な API だけ提供する」** 中間層として存在します。

```
[あなたのアプリ]
       ↓ システムコール
[ OS (カーネル) ]
       ↓ ドライバ
[ハードウェア (CPU/メモリ/ディスク/NIC)]
```

## 3-2. プロセス — 「動いているプログラム」の単位

**プロセス** とは、**実行中のプログラム1つ** のこと。
あなたが Chrome、VS Code、Slack を起動すると、それぞれ別のプロセスとして動いています。

OS から見るとプロセスはざっくり次のセットです。

- プログラムの命令列 (機械語)
- 使っているメモリ領域
- CPU レジスタの状態
- 開いているファイルやネットワーク接続
- プロセス ID (PID)

macOS なら `ps aux` や Activity Monitor、Windows なら タスクマネージャ で今動いているプロセスが見られます。

### プロセスは「自分が CPU を独占している」つもりでいる

実は CPU の数 (コア数) より、動いているプロセスの数のほうが多いのが普通です。
ではどう共存しているのか — OS が **超高速でプロセスを切り替えている** のです。これを **コンテキストスイッチ** と呼びます。

例えるなら、4つの口がある火元 (4コア CPU) で、料理人 (OS) が1秒間に何十回もフライパンを入れ替えている状態。各料理 (プロセス) からは「自分だけが火を使っている」ように見えます。

## 3-3. スレッド — プロセスの中の並行作業者

プロセスの中をさらに細かく分けたのが **スレッド** です。

- プロセス: 独立したメモリ空間を持つ。互いに干渉しない
- スレッド: 同じプロセス内で **メモリを共有しながら** 並行に走る

ブラウザを例にすると、

- プロセス1: タブA (独自のメモリ空間)
- プロセス2: タブB (独自のメモリ空間)
- プロセス2 の中で → スレッドX (UI 描画) と スレッドY (通信)

メモリを共有すると速いですが、**同時に同じ変数を書き換えると壊れる** という問題が生まれます (= 競合状態 / race condition)。これがマルチスレッドの難しさで、第8章のパラダイムや第18章のテストでも触れます。

## 3-4. メモリ管理 — 仮想メモリという嘘

物理メモリは限りがあります。でもプロセスは何個も同時に動きたい。
OS はこのギャップを **仮想メモリ (virtual memory)** という嘘で埋めます。

各プロセスには「お前専用に 0番地から広大なメモリ空間があるよ」と OS が嘘をつきます。実際には OS と CPU が裏で次のことをしています。

- 仮想アドレスを実物理アドレスに変換 (ページテーブル)
- メモリが足りなくなったら、使っていないページを **ストレージに退避** (スワップ)

このおかげでプロセス同士は互いのメモリを見られず、独立性が保てます。**「セグメンテーション違反 (segfault)」** は、自分に許されていない仮想アドレスを触ろうとして OS に止められた結果です。

:::message
仮想メモリは速度のためではなく、**保護と独立性** のための仕組み。スワップが発生し始めるとアプリは急に遅くなります (ストレージは メモリの 1000倍 遅いから)。
:::

## 3-5. ファイルシステム — ストレージを使いやすくする層

ストレージは物理的には「セクタの並んだ円盤やフラッシュメモリ」でしかありません。
それを **「フォルダの中にファイルが入っている」** という構造に見せかけているのが **ファイルシステム** です。

代表的なファイルシステム:

| OS | 代表 |
|----|------|
| Linux | ext4, xfs, btrfs |
| macOS | APFS |
| Windows | NTFS |
| USBメモリ等 | FAT32, exFAT |

ファイルシステムが提供するのは:

- **階層構造** (フォルダ/ディレクトリ)
- **ファイル名 → 物理位置** の変換
- **権限管理** (誰が読み書きできるか)
- **メタデータ** (作成日時、サイズなど)

### パーミッション (権限)

Linux/macOS では `rwx` (read / write / execute) という3つの権限を、**所有者・グループ・その他** の3者ごとに設定できます。

```bash
$ ls -l
-rw-r--r--  1 hikaru staff  1024 May 31 23:00 memo.txt
```

`-rw-r--r--` の意味は、

- 所有者: rw- (読み書き)
- グループ: r-- (読みのみ)
- その他: r-- (読みのみ)

Web サーバーが間違って `chmod 777` で運用されると、世界中の誰でも書き込める脆弱な状態になります。第17章のセキュリティでも再登場します。

## 3-6. システムコール — アプリと OS の窓口

アプリが OS に「ファイル開いて」「メモリちょうだい」「ネットに送って」と頼む手段が **システムコール** です。Python の `open()` や JavaScript の `fetch()` も、最終的にはこのシステムコールに変換されます。

代表的なもの:

| システムコール | やること |
|--------------|---------|
| `open` / `close` | ファイルを開く/閉じる |
| `read` / `write` | 読む/書く |
| `fork` / `exec` | 新しいプロセスを作る/別プログラムに置き換える |
| `mmap` | ファイルやメモリをマッピング |
| `socket` / `connect` | ネットワーク接続 |

普通のアプリ開発で直接書くことは少ないですが、「あなたが書く高級な関数の下にはこれがある」と知っておくと、ライブラリのエラーメッセージが急に読めるようになります。

## 3-7. カーネルモード と ユーザーモード

CPU には **2つの動作モード** があります。

- **ユーザーモード**: アプリが動くモード。できることが限られている
- **カーネルモード**: OS の中核 (カーネル) が動くモード。何でもできる

なぜ分けるかというと、**バグや悪意のあるアプリが OS やハードウェアを壊さないように** するためです。アプリが「ファイルを書きたい」と思っても直接ディスクを叩けない。必ずシステムコールで OS にお願いし、OS がカーネルモードに切り替えて処理する。

このモード切り替え自体にも時間がかかるので、**「I/O が多いプログラムはモード切り替え (システムコール) も多くて遅くなる」** という現場の感覚が生まれます。

## まとめ

- OS は **ハードの面倒ごとを引き受け、アプリには綺麗な API を提供する** 中間層
- **プロセス** = 実行中のプログラム1つ。コンテキストスイッチで並行動作を実現
- **スレッド** = プロセス内でメモリを共有する並行作業者。速いが競合に注意
- **仮想メモリ** = 各プロセスに独立した広大なメモリを錯覚させる仕組み。保護のため
- **ファイルシステム** = 物理ストレージを階層構造として見せる層。パーミッション付き
- **システムコール** = アプリが OS に仕事を頼む窓口
- **ユーザーモード / カーネルモード** で安全性を担保している

次の章では、あなたが OS と対話する一番素直な手段 — **シェルとコマンドライン** に進みます。
