## この章のゴール

「テーブルをどう切るか」「何を1つのドキュメントにまとめるか」 — DB の物理的な使い方の前に、**現実世界の概念をどうデータ構造に落とすか** が決定的に重要です。

ここで失敗すると、後から「なぜか機能追加のたびに苦しい」「データが矛盾しがち」「クエリが複雑」というアプリになります。

この章では **エンティティ・正規化・ER図・履歴設計・論理削除** などを扱います。

## 16-1. データモデリングの3層

設計は通常3つの層に分けて考えます。

| 層 | 何を決める |
|----|----------|
| **概念モデル** | 業務に出てくる物 (ユーザー、注文、商品...) と関係性 |
| **論理モデル** | テーブル/コレクション、列、関係 |
| **物理モデル** | DBエンジン固有の型、インデックス、パーティション |

新人がよくやる失敗は、概念モデルを飛ばして **いきなり SQL を書く** こと。
紙の上で「何があって何とつながっているか」を整理するステップを軽視すると、後で必ず手戻りします。

## 16-2. エンティティと属性

業務に登場する **「物」** を **エンティティ** と呼びます。

例: EC サイトのエンティティ
- ユーザー
- 商品
- カート
- 注文
- レビュー

各エンティティには **属性** (=DBで言う列) があります。

```
ユーザー: id, 名前, メール, パスワードハッシュ, 登録日時
商品:     id, 名前, 価格, 在庫, カテゴリ
注文:     id, ユーザーid, 合計金額, 注文日時, ステータス
```

## 16-3. 関係 (リレーションシップ)

エンティティ同士のつながり。3パターン。

| 関係 | 例 |
|------|---|
| 1対1 | ユーザー ⟷ プロフィール (詳細情報) |
| 1対多 | ユーザー ⟷ 注文 (1人が複数注文) |
| 多対多 | 商品 ⟷ タグ (1商品に複数タグ、1タグに複数商品) |

多対多は **中間テーブル** で表現 (第13章)。

## 16-4. ER図 — 関係の地図

エンティティと関係を可視化したのが **ER図 (Entity-Relationship Diagram)**。

```
[User] 1───* [Order] *───* [Product]
                              ↑
                             *
                              [Tag]
```

「1」「*」「0..1」のような **カーディナリティ** (どれだけの数を持ちうるか) を書くのが定石。
チームで設計を共有するときに必須のドキュメント。dbdiagram.io や draw.io、Mermaid で簡単に書けます。

## 16-5. 正規化 — 重複を排除する設計手法

データの **冗長性 (同じ情報の重複)** を排除する手順。1〜3 まで覚えれば現場では十分。

### 第1正規形 (1NF)

各セルに **単一の値** を入れる。配列・リストを直接入れない。

❌ ダメな例:

| user_id | name | tags |
|---------|------|------|
| 1 | Hikaru | "python, iot, swift" |

✅ 1NF:

| user_id | name |
|---------|------|
| 1 | Hikaru |

| user_id | tag |
|---------|-----|
| 1 | python |
| 1 | iot |
| 1 | swift |

### 第2正規形 (2NF)

**主キーの一部に依存する列を分離** する。

❌: `(注文id, 商品id, 商品名, 商品単価, 個数)` のうち、商品名と単価は商品idだけで決まる
✅: 注文明細から商品名・単価を別テーブルに

### 第3正規形 (3NF)

**主キー以外の列に依存する列を分離** する。

❌: `(社員id, 部署id, 部署名)` のうち、部署名は部署idだけで決まる
✅: 社員テーブルと部署テーブルを分ける

### 正規化の効用と弊害

| | 利点 | 欠点 |
|--|------|------|
| 正規化 | データの重複なし、整合性◎ | JOIN が多くなる |
| 非正規化 | クエリが速い、シンプル | データが重複、整合性管理が大変 |

OLTP (普通のアプリ) は正規化、OLAP (集計・分析) は非正規化、というのが大まかな目安。

## 16-6. 履歴・スナップショット設計

「ユーザーの住所」を1列で持つと、引っ越し後に過去の注文の配送先が分からなくなります。

### 1. 上書き型 (素朴だが履歴が消える)

```
users(id, name, address)
```

### 2. 履歴テーブル方式

```
users(id, name, current_address_id)
addresses(id, user_id, address, valid_from, valid_to)
```

### 3. 注文に住所をコピー (スナップショット)

```
orders(id, user_id, shipping_address, ...)  -- 注文時の住所を保存
```

実務では **注文・請求書・領収書のような「ある時点の事実」はスナップショットで持つ** のが定石。
ユーザーの最新情報を引っ張ると履歴がズレるバグになります。

## 16-7. 論理削除 vs 物理削除

「削除されたデータ」をどう扱うか。

### 物理削除

`DELETE` で本当に消す。
シンプル。容量も食わない。

### 論理削除 (ソフトデリート)

`deleted_at` 列を立てて、消えたことにする。

```sql
SELECT * FROM users WHERE deleted_at IS NULL;
```

| | 物理 | 論理 |
|--|------|------|
| 復元 | 不可 | 可能 |
| 容量 | ◎ | △ |
| クエリの複雑さ | ◎ | × (毎回 IS NULL が要る) |
| 監査・規制 | 弱い | 強い |

「**論理削除すれば安心**」と無条件に採用すると、`WHERE deleted_at IS NULL` を忘れた瞬間に消したはずのデータが現れます。**目的に応じて選ぶ**。

## 16-8. ID 設計 — 連番 vs UUID

| 方式 | 利点 | 欠点 |
|------|------|------|
| 連番 (BIGINT AUTO_INCREMENT) | 短い、ソートしやすい | 推測されやすい、分散で衝突 |
| UUID v4 | グローバル一意、推測不能 | 長い、ソートできない |
| ULID / UUID v7 | ソート可能 + 一意 | やや新しめ |

**API で URL に出る ID は推測されないものにする** (UUID または ハッシュ ID)。
ユーザーIDが `1, 2, 3...` だと「全ユーザーを順に総当たり」できてしまう。

## 16-9. タイムスタンプ 3点セット

ほぼ全テーブルに付けてほしい列:

```sql
created_at TIMESTAMP NOT NULL DEFAULT NOW(),
updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
deleted_at TIMESTAMP NULL    -- 論理削除を使う場合
```

これがないと「いつ作られた」「いつ最後に変わった」が分からず、デバッグも分析もできません。

## 16-10. NoSQL 設計の発想 — 「クエリから逆算」

RDB は **データ構造を先に決め**、後で SQL でなんとかします。
NoSQL は逆で、**「どんなクエリが来るか」を先に決め、その形に合うようにデータを持つ** のが基本。

例: メッセージアプリ
- クエリ: 「あるルームの最新メッセージ100件」
- NoSQL 設計: ルームIDをパーティションキー、タイムスタンプをソートキー

「**正規化しない、JOIN しない、検索クエリパターンに最適化する**」 — 逆の発想に慣れる必要があります。

## 16-11. イベントソーシング — 状態でなくイベントを保存

「最新状態」だけ持つのではなく、「**起きたことの履歴**」をすべて保存する設計。

```
イベント:
- UserRegistered(id=1, name="Hikaru")
- EmailChanged(id=1, email="new@example.com")
- UserDeleted(id=1)
```

そこから現在状態を再構築 (リプレイ) する。

利点:
- 過去の状態に戻せる
- 監査ログそのもの
- イベント駆動アーキテクチャと相性◎

弊害:
- 実装複雑、学習コスト高
- 普通の CRUD には過剰

知識として持っておくと、「履歴が命の業務 (金融、医療)」で力を発揮します。

## 16-12. 設計の評価基準

良いデータモデルかどうかは、次で測れます。

1. **業務用語と一致しているか** (UserRegistration を Reg と略さない)
2. **新機能の追加でテーブル構造を破壊しないで済むか**
3. **クエリが素直に書けるか**
4. **データの矛盾が原理的に起きないか**
5. **過去のデータが残るべきものは残るか**

3つ以上に「YES」と言えれば現場でかなり戦えます。

## まとめ

- 設計は **概念 → 論理 → 物理** の3層で進める。いきなり SQL を書かない
- エンティティ・属性・関係 (1対1, 1対多, 多対多) を ER図で見える化
- **正規化 (1NF/2NF/3NF)** で重複を排除。集計用には非正規化も
- 過去の事実 (注文、請求) は **スナップショット** で保存
- 論理削除と物理削除を **目的で使い分ける**
- ID は **連番 / UUID / ULID** をユースケースで選ぶ
- **created_at / updated_at / deleted_at** はほぼ全テーブルに
- NoSQL は **クエリから逆算** が基本姿勢
- **イベントソーシング** は履歴が命の業務で強力
- 良い設計は「業務用語と一致」「拡張に強い」「クエリが素直」
