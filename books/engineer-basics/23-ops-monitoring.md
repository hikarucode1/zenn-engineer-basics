## この章のゴール

「**動いた」と「動き続けている」は別の世界**です。
本番運用が始まった瞬間から、24時間ユーザーが触り、障害が起き、トラフィックが波打ちます。

この章では、Google が体系化した **SRE (Site Reliability Engineering)** の発想を借りつつ、メトリクス・アラート・SLO・障害対応・キャパシティ・オンコールを扱います。

## 23-1. SRE — 「運用をソフトウェアエンジニアリングで解く」

Google が2003年頃に始めた職種・思想。

要点:
- **インフラ・運用も「コードで」自動化する**
- **品質目標 (SLO)** を数値で決める
- **障害ゼロは目指さない** (ゼロにするとイノベーションが止まる)
- 過剰な手作業を減らし、**Toil (退屈な反復作業) を仕組みで潰す**

「障害ゼロでなく、ユーザー体験が許容できるレベルで保つ」が思想の核心です。

## 23-2. SLI / SLO / SLA — 3つの指標

混同しやすい3語。

| 略 | 名前 | 意味 |
|----|-----|------|
| **SLI** (Service Level Indicator) | 実測値 | 「直近30日の API 成功率は 99.92%」 |
| **SLO** (Service Level Objective) | 内部目標 | 「99.9% を目標にする」 |
| **SLA** (Service Level Agreement) | 顧客との契約 | 「99.5% 下回ったら返金」 |

SLI = 測る、SLO = 目指す、SLA = 約束する。
SLO は SLA より厳しく設定するのが定石。

### エラーバジェット

100% - SLO = **障害してよい量**。
例: SLO 99.9% なら 1ヶ月で 43分まで障害OK。

この「許される時間」をデプロイのリスクテイクに使うのが SRE 流。**バジェットが余っていれば積極的に新機能デプロイ**、使い切ったら **安定化フェーズ**。

## 23-3. 観測可能性 (Observability) 3点セット

第19章でも触れた3つを統合管理。

| 種類 | 答えられる質問 | ツール例 |
|------|---------------|---------|
| **メトリクス** | 何がどれだけ起きた? | Prometheus, Datadog, CloudWatch |
| **ログ** | 具体的に何があった? | Elasticsearch, Loki, CloudWatch Logs |
| **トレース** | 1リクエストの経路は? | Jaeger, AWS X-Ray, Datadog APM |

### メトリクスの代表例

| 種類 | 例 |
|------|----|
| **RED**: Rate / Errors / Duration | API 監視で必須 |
| **USE**: Utilization / Saturation / Errors | サーバー監視で必須 |
| Apdex | ユーザー満足度の指標 |

**4 Golden Signals** (Google SRE 本): Latency / Traffic / Errors / Saturation — これだけ押さえれば 80% は判断できる。

## 23-4. アラート設計

「アラートが多すぎる → 無視されて慣れる → 本物を見逃す」が現場あるある。

良いアラートの条件:
1. **症状ベース** (ユーザーが困っていることに対して鳴らす)
2. **アクションがある** (鳴ったら何をすればいいか明確)
3. **誤検知が少ない**
4. **抑制 (重複・連鎖) の仕組みがある**

❌ ダメな例: 「CPU が 80% 超えた」(原因系。ユーザーが困っているか不明)
✅ 良い例: 「過去5分のAPI成功率が99%を下回った」(症状系)

## 23-5. 障害対応 (Incident Response)

障害が起きたときの動き方。

1. **検知**: アラート、ユーザー報告
2. **トリアージ**: 影響範囲・優先度を決める
3. **緩和**: まず止血 (ロールバック、サーキットブレーカー)
4. **復旧**: 通常状態に戻す
5. **ポストモーテム**: 振り返り

### 役割分担 (Incident Command)

大規模障害ではチームで動く。
- **Incident Commander**: 全体指揮
- **Communication Lead**: 内外への連絡
- **Operations Lead**: 復旧作業

「**まず止血、原因究明は後**」が鉄則。長時間ユーザーが困っているのが一番ダメ。

### ポストモーテム (Postmortem)

障害の振り返り。重要なのは:

- **責めない (Blameless)** — 個人を犯人にしない
- **再発防止に焦点**
- **タイムラインを残す** (いつ何が起きたか)
- **5 Whys** (5回 Why を繰り返して根本原因へ)

文化として **「誰も罰されない振り返り」** を作れる組織だけが、技術的負債と障害から学び続けられます。

## 23-6. キャパシティプランニング

「**ユーザー10倍になったら持つ?**」を考える作業。

ステップ:
1. 現在の使用率 (CPU / メモリ / DB接続) を測る
2. ピーク時とアイドル時の比を見る
3. 将来予測 (ビジネス側に聞く)
4. ボトルネックを特定 (CPU? DB? ネットワーク?)
5. スケール方式を決める (垂直 / 水平)
6. オートスケーリングの設定

### 垂直スケール vs 水平スケール

| | 垂直 | 水平 |
|--|------|------|
| 何をする | 1台を強くする | 台数を増やす |
| 簡単さ | ◎ | △ (アプリの設計が要る) |
| 上限 | あり (1台の物理上限) | 事実上無限 |
| DB | 普通こっち | 工夫が要る (シャーディング) |

ステートレスなアプリは水平、DB は垂直、というのが伝統的な感覚。

## 23-7. オートスケーリング

「**トラフィックに応じてサーバー台数を自動で増減**」する仕組み。

トリガー例:
- CPU 使用率
- リクエスト数
- キューの長さ
- カスタムメトリクス

```
平日昼:    5台
夜23時:    2台 (自動縮小)
キャンペーン日: 20台 (自動拡張)
```

設計のコツ:
- **スパイクに対して反応が遅い** ことを考慮 (起動に数分かかる)
- **コールドスタートを許容する設計** (キャッシュウォームアップ)
- 縮小は緩やかに (急に縮めて余分なリクエストが来ると詰まる)

## 23-8. リトライ・タイムアウト・サーキットブレーカー

ネットワーク越しの呼び出しに必須のパターン。

### タイムアウト

「**いつまで待つか**」を明示する。
タイムアウトを設定しないと、外部APIが応答しない時にスレッドが詰まり全体が死ぬ。

### リトライ

「**失敗したら何回までやり直すか**」。
- リトライ回数の上限
- **指数バックオフ** (1秒 → 2秒 → 4秒 …)
- ジッタ (微小なランダム揺れ) を入れて全クライアントが同時に再試行する事故を防ぐ

### サーキットブレーカー

「**呼び先が壊れているなら、しばらく呼ぶのをやめる**」パターン。
- Closed (通常)
- Open (失敗多発、呼ばずに即エラー)
- Half-Open (試しに少しだけ呼んで様子見)

電気のブレーカーが落ちるのと同じ。これがないと **障害が連鎖** して全部巻き込まれます。

## 23-9. バックアップとディザスタリカバリ

「**データが消えたらどうするか**」を考えていますか?

### バックアップの2軸

- **RPO (Recovery Point Objective)**: どこまで遡って失っていいか (1時間前まで?)
- **RTO (Recovery Time Objective)**: 何時間以内に復旧するか

### 3-2-1 ルール

- **3** つのコピー
- **2** つの異なる媒体
- **1** つはオフサイト (別地域)

クラウドの S3 は耐久性が高いとはいえ、**人為的削除** には弱い。バージョニング・別アカウントへのコピーで二重に守る。

### リカバリ訓練

「バックアップを取っている」と「**復元できる**」は別。**実際に試す**。
復元したことがないバックアップは存在しないのと同じ、という格言があります。

## 23-10. オンコール (On-call) 文化

「障害時に呼ばれる当番」。

健全な運用:
- **ローテーションする** (一人に集中させない)
- **エンジニアが自分のサービスを当番する** (運用専門チームではなく)
- **オンコール手当** が出る
- **無理なシフトを組まない**

新人時代から「**自分のコードは自分で運用する**」感覚を持つと、品質が一気に上がります。「**you build it, you run it**」(by Werner Vogels, AWS CTO) という言葉が有名。

## 23-11. Toil (退屈な反復作業) を潰す

SRE の重要概念。

Toil の特徴:
- 手作業
- 反復的
- 自動化可能
- 価値が増えない (定型タスク)

例: 「毎週手動でログを集計して報告書を出す」
→ ジョブで自動化。エンジニアは別の仕事へ。

**Toil を業務時間の50%以下に保つ** のが SRE の目標。

## 23-12. ダッシュボードと「いつ何を見るか」

監視データはダッシュボードで可視化。

- **オンコール用**: 4 Golden Signals が一目で
- **製品オーナー用**: SLO の達成度、ユーザー指標
- **障害時用**: 関連サービスの状態が並ぶ

「**毎朝5分だけダッシュボードを眺める**」習慣がチームに根付くと、異常検知が早くなります。

## まとめ

- **SRE** は「運用をソフトウェアエンジニアリングで解く」思想
- **SLI / SLO / SLA**: 測る・目指す・約束する
- **エラーバジェット** で安定性と新機能のバランスを取る
- 観測 3点: **メトリクス・ログ・トレース**
- 4 Golden Signals: **Latency / Traffic / Errors / Saturation**
- アラートは **症状ベース** で、誤検知を減らす
- 障害対応は **止血 → 復旧 → ポストモーテム (blameless)**
- スケールは **垂直 / 水平**、オートスケーリングで自動化
- **タイムアウト・指数バックオフ・サーキットブレーカー** で障害連鎖を防ぐ
- バックアップは **3-2-1 ルール**、復元を必ず試す
- **You build it, you run it** — 自分のコードは自分で運用
- **Toil** を自動化で50%以下に保つ
