## この章のゴール

プログラムが扱うデータは「変数1つ」では収まりません。
ユーザーリスト、商品一覧、メッセージ履歴 — どれも **多数のデータをまとめて、効率よく出し入れする** 必要があります。

その「まとめ方の設計図」が **データ構造** です。
この章では、現場でほぼ毎日使う基本構造 — 配列、連結リスト、スタック、キュー、ハッシュマップ、木、グラフ — を一通り見ていきます。

## 6-1. なぜデータ構造を学ぶのか

「Python の list を使えば全部済むのでは?」と思った人。半分正解、半分大事なポイントを逃しています。

list を使うときに **「これは追加が速い? 検索が速い? メモリは?」** を判断できないと、データが増えたときに突然遅くなって原因が分からないエンジニアになります。

データ構造を学ぶ = **「目的に対して最適な箱を選べるようになる」** です。

## 6-2. 配列 (Array) — 連続したメモリの並び

最も基本的なデータ構造。**メモリ上に連続して並んだ箱** です。

```
インデックス: 0   1   2   3   4
            [10][20][30][40][50]
```

特徴:
- インデックス参照が **O(1)** = 一瞬
- 要素の追加・削除 (途中) が **O(n)** = 全部ずらす必要あり
- メモリは事前確保 (固定長 or 動的拡張)

C の配列は固定長、Python の list、JS の Array は動的に伸びるので「動的配列」と呼ばれます。

## 6-3. 連結リスト (Linked List) — ノードを矢印でつなぐ

各要素 (ノード) が「値」と「次のノードへの参照」を持つ構造。

```
[10|→][20|→][30|→][40|→]null
```

特徴:
- 途中への挿入・削除が **O(1)** (矢印を付け替えるだけ)
- インデックス参照が **O(n)** (最初から順にたどる)
- メモリは飛び飛びでOK

「途中挿入が頻繁」なときは連結リスト。逆に「ランダムアクセスが多い」なら配列。
ほとんどの現代言語の標準 list は実は動的配列で、純粋な連結リストはあまり使われませんが、概念としては必須知識です。

## 6-4. スタック (Stack) — 後入れ先出し (LIFO)

「**Last In, First Out**」 — 一番最後に入れたものから取り出す。

```
push(1) push(2) push(3)  → [1, 2, 3]
pop()                     → 3 が出る (残り [1, 2])
```

例えで言うと、**洗った皿を重ねていく**。一番上の皿から使う。

使いどころ:
- 関数の呼び出し履歴 (= コールスタック)
- 戻る操作 (ブラウザの「戻る」、Ctrl+Z)
- 括弧の対応チェック

## 6-5. キュー (Queue) — 先入れ先出し (FIFO)

「**First In, First Out**」 — 最初に入れたものから取り出す。

```
enqueue(1) enqueue(2) enqueue(3) → [1, 2, 3]
dequeue()                          → 1 が出る (残り [2, 3])
```

例えで言うと、**レジの行列**。並んだ順に対応。

使いどころ:
- タスクキュー (順に処理したいジョブ)
- メッセージキュー (RabbitMQ, SQS など)
- 幅優先探索 (BFS)

## 6-6. ハッシュマップ (HashMap / Dictionary) — キー→値の辞書

「キー」を渡すと「値」が返ってくる、**辞書のような構造**。

```python
user = {"name": "Hikaru", "age": 25}
user["name"]  # → "Hikaru"
```

特徴:
- 検索・挿入・削除がすべて **平均 O(1)**
- キーから ハッシュ関数で位置を計算する
- メモリは多めに使う

これがすべての言語に標準搭載されている理由は、**現場で最も使うデータ構造だから**。
- ユーザー ID → ユーザーオブジェクト
- 商品コード → 在庫数
- セッションID → ログイン情報

### ハッシュ衝突

別々のキーから同じ位置が算出されることがあります。これを **衝突** と言い、内部的にチェイン (連結リスト) や別の手法で解決します。これが起きすぎると O(1) が O(n) に落ちるので、ハッシュ関数の設計が重要です。

## 6-7. 集合 (Set) — 重複なしのコレクション

ハッシュマップから「値」を抜いて「キーだけ」にしたもの。

```python
visited = set()
visited.add("user1")
visited.add("user1")  # 無視される
"user1" in visited  # True
```

「**いるかいないか**」を高速に判定したいとき使います。重複除去、訪問済みノードの記録など。

## 6-8. 木 (Tree) — 階層構造

**親子関係のあるノード** の集まり。

```
        [root]
       /      \
    [A]        [B]
   /   \         \
 [C]   [D]       [E]
```

代表例:
- **ファイルシステム** (フォルダ階層)
- **DOM** (HTMLの構造)
- **AST** (プログラムの構文木)
- **B-Tree** (DBのインデックス)

### 二分探索木 (BST)

各ノードが「左の子は自分より小さい、右の子は自分より大きい」というルールを守る木。
探索・挿入・削除がすべて **平均 O(log n)** という非常に効率的な構造で、DB のインデックスや TreeMap の基礎になっています。

## 6-9. グラフ (Graph) — つながりの構造

**ノードと辺 (エッジ)** で表現される、最も一般的な構造。木はグラフの特殊形 (閉じた経路がない)。

代表例:
- **SNS のフォロー関係**
- **地図 (道路ネットワーク)**
- **依存関係** (npm パッケージ、Make ターゲット)
- **状態遷移**

実装は **隣接リスト** か **隣接行列** が一般的。グラフを扱うアルゴリズム (BFS, DFS, ダイクストラ法など) は次章で触れます。

## 6-10. 「どれを選ぶか」 ガイド

| やりたいこと | 選ぶデータ構造 |
|-------------|---------------|
| インデックスで素早く参照 | **配列** |
| 順番を保ちつつキューで処理 | **キュー** |
| 関数の呼び出し履歴 | **スタック** |
| キーで素早く値を引く | **ハッシュマップ** |
| 重複を許さず存在判定 | **集合** |
| 階層を表す | **木** |
| 何でもつながるネットワーク | **グラフ** |

迷ったらまずハッシュマップを試す、というのが現場の経験則です。

## 6-11. 計算量との対応表

各データ構造の操作にかかる時間。これは次章の「アルゴリズム」で使うので軽く目を通しておいてください。

| 構造 | 参照 | 検索 | 挿入 | 削除 |
|------|------|------|------|------|
| 配列 | O(1) | O(n) | O(n) | O(n) |
| 連結リスト | O(n) | O(n) | O(1) | O(1) |
| スタック | - | - | O(1) | O(1) |
| キュー | - | - | O(1) | O(1) |
| ハッシュマップ | - | O(1) | O(1) | O(1) |
| 二分探索木 | - | O(log n) | O(log n) | O(log n) |

## まとめ

- データ構造は **「目的に合った箱」を選ぶための設計図**
- **配列**: 連続メモリ、参照が速い
- **連結リスト**: 飛び飛びメモリ、途中挿入が速い
- **スタック (LIFO) / キュー (FIFO)**: 順番を制御する仕組み
- **ハッシュマップ**: キー→値、O(1) で出し入れ。現場の主力
- **集合**: 重複なしの存在判定
- **木**: 階層構造。BST は O(log n)
- **グラフ**: 一般的なつながり。SNS や地図の表現
- どの操作が何 O(n) になるかを意識すると、性能の予測が立てられる

次の章では、これらの構造を **どう動かすか** — つまりアルゴリズムの基本に進みます。
