## この章のゴール

第1章でコンピュータの登場人物を眺めました。次に気になるのは、**「彼らは結局なにを運んでいるのか?」** という問いです。

答えはたった2文字、**0 と 1** です。
文字も画像も音声も動画も、コンピュータの中ではすべて 0 と 1 の並びになっています。この章では、その「世界共通の言葉」がどう設計されているかを学びます。

## 2-1. なぜコンピュータは 0 と 1 しか使わないのか

理由はシンプルで、**電気のオン/オフが一番安定して区別できるから** です。

「3.7V なら 3、1.2V なら 1」のような細かい区別をしようとすると、ノイズや温度変化ですぐ間違えます。一方「0V 付近 = 0、5V 付近 = 1」のような2値なら、多少電圧が揺れてもどちらかに倒せます。

この **0 か 1 かの最小単位** を **bit (ビット)** と呼びます。

## 2-2. bit と byte

- **bit**: 0 か 1 の1桁
- **byte (バイト)**: bit を 8 個集めたもの (= 8bit)

なぜ 8 なのかというと、歴史的経緯です。「英数字1文字をだいたい1単位で扱える便利な大きさ」として 8bit = 1byte が業界標準になりました。

byte の上位単位は次の通りです。

| 単位 | サイズ | だいたいの目安 |
|------|--------|---------------|
| 1 KB (キロ) | 約 1,000 byte | 短い文章 |
| 1 MB (メガ) | 約 100万 byte | 1分の音楽 / 写真1枚 |
| 1 GB (ギガ) | 約 10億 byte | 映画1本 |
| 1 TB (テラ) | 約 1兆 byte | 映画1000本 |

:::message
厳密には KB は 1024 byte (2進法) を指すこともあれば 1000 byte (10進法) を指すこともあります。ストレージの広告は 1000、メモリは 1024 を使うことが多く、これが「1TB の HDD を買ったのに 931GB しか表示されない」現象の原因です。
:::

## 2-3. 2進数の数え方

私たちが普段使う **10進数** は、0〜9 の10種類の数字を使い、桁が上がるごとに10倍されます。

2進数は **0〜1 の2種類** を使い、桁が上がるごとに2倍されます。

| 10進 | 2進 |
|------|-----|
| 0 | 0 |
| 1 | 1 |
| 2 | 10 |
| 3 | 11 |
| 4 | 100 |
| 5 | 101 |
| ... | ... |
| 10 | 1010 |

`1011` (2進) = `1×8 + 0×4 + 1×2 + 1×1` = **11** (10進)、というように、右から 1, 2, 4, 8, 16... と重みをかけて足すだけです。

### 16進数 (hex) もよく出てくる

2進数は人間には読みにくいので、現場では **16進数** (0〜9, a〜f) もよく使われます。理由は **4bit がちょうど 16進1桁に対応する** ためです。

```
2進: 1111 1010   →  16進: FA  →  10進: 250
```

色コード `#FF0000` (赤) や、メモリアドレス `0x7ffee000`、ハッシュ値などは全部16進数表記です。

## 2-4. 負の数を表す — 2の補数

「2進数だと負の数はどう表すんだろう?」という疑問が出てきます。
答えは **2の補数 (two's complement)** という仕組みで、ほぼすべての CPU がこれを使っています。

ざっくり言うと:

1. 一番左の bit を **符号ビット** とする (1 なら負)
2. 負の数は「ビットを反転して 1 を足す」ことで表す

8bit の例:

| 値 | 2進表現 |
|----|--------|
| 5 | `00000101` |
| -5 | `11111011` |

この仕組みのおかげで、CPU は **加算回路だけで引き算も処理できる** という非常にエレガントな設計になっています。

## 2-5. 小数の表現 — 浮動小数点

整数と違って、小数 (0.1 や 3.14) は2進数で正確に表現できないことがあります。

`0.1` を2進数にすると `0.00011001100110011...` と無限に続きます。これを有限桁で打ち切るので、**わずかな誤差** が生まれます。

```javascript
0.1 + 0.2 // → 0.30000000000000004
```

これはバグではなく、**IEEE 754 という浮動小数点規格** に従った正しい挙動です。お金の計算で `float` を使うとずれるのはこのため。

:::message
お金や正確さが要る計算は整数 (円単位) か Decimal 型 を使う、と覚えておきましょう。これは第13章 DB のときにもう一度出てきます。
:::

## 2-6. 文字の表現 — 文字コード

数字や記号は数値に対応させればいいですが、**文字** はどう扱うのでしょうか。
答えは「文字に番号を振って、その番号を保存する」というルールを決めること。これが **文字コード** です。

### ASCII (アスキー)

英数字と記号、合計128種類に番号を振った最古の規格。

| 文字 | コード (10進) | コード (16進) |
|-----|--------------|---------------|
| `A` | 65 | 0x41 |
| `a` | 97 | 0x61 |
| `0` | 48 | 0x30 |
| 改行 | 10 | 0x0A |

ASCII は 1 文字 = 1 byte で済む代わりに、**日本語などの非ラテン文字を扱えません**。

### Shift_JIS と EUC-JP

ASCII を拡張して日本語を扱えるようにした規格。Windows のメモ帳が長年使ってきたのは Shift_JIS です。Mac/Linux 系では EUC-JP も使われました。

これらは「日本語専用」なので、中国語や韓国語、絵文字を混ぜると破綻します。

### Unicode と UTF-8

そこで生まれたのが **Unicode** = 「世界中の文字に1つの番号体系を割り当てる」プロジェクト。`あ` は `U+3042`、`漢` は `U+6F22` のように番号 (コードポイント) を与えます。

そして、その番号を **実際のバイト列に変換する方式** が **UTF-8** などです。

| 規格 | 1文字あたりのバイト数 |
|------|--------------------|
| UTF-8 | 1〜4 byte (ASCII は1byte、日本語は3byte) |
| UTF-16 | 2 か 4 byte |
| UTF-32 | 常に 4 byte |

**現代のほぼすべての Web、ファイル、API は UTF-8 がデフォルト**。迷ったら UTF-8 を選んでおけば、ほぼ間違いありません。

### 文字化け

「文字化け」とは、**保存したときの文字コードと、読み込むときの文字コードがズレた状態** です。

```
保存: UTF-8 で「こんにちは」
読み込み: Shift_JIS だと解釈 → 「縺薙s縺ｫ縺｡縺ｯ」
```

新人エンジニアが最初にぶつかる罠の代表格ですが、原理はこれだけ。

## 2-7. 画像・音声・動画も全部 0 と 1

- **画像**: ピクセル (= 小さな点) を縦横に並べ、各ピクセルの色を RGB の数値で表す
- **音声**: 空気の振動を一定間隔で数値に変換する (サンプリング)
- **動画**: 画像を毎秒 30〜60 枚並べる + 音声

つまりあらゆるデータは「数値の並び」に還元され、その数値が 2進数になり、最後は電気のオン/オフになる、というのがコンピュータのすべてです。

## まとめ

- コンピュータが扱うのは **0 と 1 だけ**。電気の安定性が理由
- **bit (1桁) → byte (8bit)** が単位の基本。KB/MB/GB/TB と上がっていく
- 2進数は重みを足すだけ。16進数は 4bit 単位で書きやすくしたもの
- 負の数は **2の補数**、小数は **浮動小数点 (IEEE 754)** で誤差が出る
- 文字は「文字 ↔ 番号」のルール = **文字コード**。**迷ったら UTF-8**
- 画像も音声も動画も結局「数値の並び」=最終的には 0/1

次の章では、これら 0/1 を実際にどう動かすか — つまり **OS (オペレーティングシステム)** の役割を見ていきます。
