## この章のゴール

「テストを書く」と聞くと、新人は「動作確認をひと手間かけて自動化するだけ」だと思いがちです。
実際は **設計の道具・改修の安全網・ドキュメント・採用市場での通貨** — テストはコードと同じくらい重要な資産です。

この章では、テストの種類・書き方・カバレッジ・モック・テスト設計の原則を学びます。

## 18-1. なぜテストを書くのか

理由は4つ。

1. **退行 (regression) を防ぐ**: 既存機能を壊さずに変更できる
2. **設計のフィードバック**: テストが書きづらいコードはたいてい設計が悪い
3. **仕様書になる**: 「このコードは何をするか」の生きたドキュメント
4. **採用評価**: テストが書けることはエンジニアとしての必須スキル

「テストを書く時間がない」が真実なのは1割。残り9割は「**テスト無しで書いたコードのデバッグに時間を吸われている**」のが現実です。

## 18-2. テストピラミッド

テストには種類があり、量のバランスがあります。

```
        /\
       /E2E\          少数・遅い・本物に近い
      /------\
     / 統合  \         そこそこ
    /----------\
   /  単体テスト \     大量・速い・安い
  /--------------\
```

- **単体テスト (Unit)**: 関数1つ、クラス1つを単独でテスト
- **統合テスト (Integration)**: 複数モジュール、DB、外部API を組み合わせる
- **E2E (End-to-End)**: ユーザー操作と同じ流れで本物のブラウザを動かす

このピラミッドの形を守るのが現代的なベストプラクティス。下が大きく、上に行くほど少なくなる。

## 18-3. 単体テスト (Unit Test)

最小単位の関数・クラスをテスト。
依存は **モック** に置き換えて、対象だけ確認する。

```python
def add(a, b):
    return a + b

# テスト
def test_add():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0
```

### AAA パターン

単体テストの定型構造。

```python
def test_user_login():
    # Arrange: 準備
    user = User(name="Hikaru", password="secret")

    # Act: 実行
    result = user.login("secret")

    # Assert: 検証
    assert result.is_success
```

`Arrange / Act / Assert` を空行で分けて書くと、後から読む人に優しい。

### 主要フレームワーク

| 言語 | テストフレームワーク |
|------|--------------------|
| Python | pytest, unittest |
| JavaScript | Jest, Vitest |
| Java | JUnit |
| Go | 標準 `testing` パッケージ |
| Ruby | RSpec, Minitest |

## 18-4. 統合テスト (Integration Test)

DB を本物使って、エンドポイントを叩く。「複数の部品が**つながっているか**」を見る。

```python
def test_create_user_endpoint(client, db):
    response = client.post("/users", json={"name": "Hikaru"})
    assert response.status_code == 201
    assert db.query(User).count() == 1
```

DB のテストは:
- **テスト用のDB を別途立てる** (Docker などで)
- **各テスト前後でリセット** (フィクスチャ、トランザクションロールバック)
- **本番DBには絶対つながない**

## 18-5. E2E テスト (End-to-End Test)

ブラウザを実際に操作してユーザーの体験を再現。

```javascript
// Playwright の例
test('ログインしてダッシュボードに飛ぶ', async ({ page }) => {
  await page.goto('https://app.example.com/login');
  await page.fill('input[name=email]', 'h@example.com');
  await page.fill('input[name=password]', 'secret');
  await page.click('button[type=submit]');
  await expect(page).toHaveURL(/dashboard/);
});
```

代表ツール:
- **Playwright** (Microsoft) — 現代の主流
- **Cypress** — JS界で人気
- **Selenium** — 老舗、レガシー

E2E は **遅い・壊れやすい** ので、最小限の主要シナリオだけに絞るのが鉄則。

## 18-6. モック・スタブ・スパイ

依存を差し替える3つの道具。

| 用語 | 何をする |
|------|---------|
| **モック (Mock)** | 振る舞いを差し替え、呼ばれたか検証 |
| **スタブ (Stub)** | 決まった値を返すだけ |
| **スパイ (Spy)** | 本物を動かしつつ呼ばれ方を記録 |

```python
# 外部API を呼ばずに済むようモック
@patch('myapp.requests.get')
def test_fetch_user(mock_get):
    mock_get.return_value.json.return_value = {"id": 1, "name": "Hikaru"}
    user = fetch_user(1)
    assert user.name == "Hikaru"
```

### モックしすぎ問題

「全部モックしたテスト」は、**本物が動くかは全く確認していません**。
**「本物に近いほど信頼度が高い」** を意識して、外部API・課金・メール送信などやむを得ない部分だけモック化するのが現代的。

## 18-7. カバレッジ

テストがコードのうち何%を実行したかの指標。

- **行カバレッジ** — 何行通ったか
- **分岐カバレッジ** — if/else の両方通ったか
- **条件カバレッジ** — 条件式の各項を検証したか

「**カバレッジ 100% = バグなし** ではない」 — これが最大の罠。テストが全行を通っていても、間違ったロジックを「正しい」と検証してしまえばバグは残ります。
**カバレッジは目安、目的化しない**。

実務感覚として、業務ロジックは 80% 以上、UI 周りは 50〜70%、というあたりが現実的なライン。

## 18-8. TDD (Test-Driven Development)

「**テスト → 実装**」の順で書く開発手法。

```
1. Red: 失敗するテストを書く
2. Green: 最小限のコードで通す
3. Refactor: コードを整理する
```

学習効果が大きく、設計力が上がる。**全部 TDD で書く必要はないが、一度は通ってみる** ことを強く勧めます。

## 18-9. プロパティベーステスト

「**こういう性質を持つはず**」をランダム入力で大量に検証するテスト。

```python
from hypothesis import given, strategies as st

@given(st.lists(st.integers()))
def test_sort_is_idempotent(lst):
    assert sorted(sorted(lst)) == sorted(lst)
```

Hypothesis (Python), QuickCheck (Haskell), fast-check (JS) などが有名。
境界値を自分で考えなくても、フレームワークが意地悪な入力を投げてくれる。

## 18-10. ミューテーションテスト

「**テストを通った状態のコードを、わざと壊してみてテストが落ちるか**」を確認する手法。
カバレッジより踏み込んだ品質指標。**テストの質**を測る。

ツール: mutmut (Python), Stryker (JS), PIT (Java)

## 18-11. テストが書きづらいコードの特徴

新人が最初にぶつかるのが「テストが書けないコード」。これは **設計の警告サイン**。

典型例:
- 関数が長すぎる (100行とか) → 分割しろ
- グローバル変数依存 → 引数で渡せ
- 外部API直叩き → インターフェースで切れ
- 副作用が多い → 純粋関数に寄せろ
- 隠れた依存 (シングルトン) → 明示的に依存注入

第8章の関数型志向、第12章の API 設計と直結する話です。

## 18-12. テスト戦略の現場感

| アプリの段階 | テスト方針 |
|-----------|----------|
| 試作 (PoC) | テストなし or 主要シナリオだけ |
| 成長期 | 単体テストを薄く、主要 E2E |
| 安定期 | 単体・統合をしっかり、E2E は主要動線 |
| 大規模・本番 | 上+ パフォーマンス、契約テスト |

「**最初から完璧にテストを書こうとしない**」のが現実的。動くコードを優先しつつ、機能が落ち着いたらテストで固める。

## まとめ

- テストは「念のため」ではなく **退行防止・設計道具・仕様書・採用通貨**
- ピラミッド: **単体 (多) > 統合 > E2E (少)**
- 単体テストは **AAA** 構造で書く
- 統合テストは **本物の DB** を使い、テスト用に分離
- E2E は Playwright が主流、最小シナリオに絞る
- **モック/スタブ/スパイ** で依存を差し替える、ただし**やりすぎない**
- カバレッジは目安、**100% = バグなしではない**
- **TDD** で設計力が上がる、一度は通ってみる価値あり
- テストが書きづらい = **設計が悪いサイン**
- 段階に応じて戦略を変える。最初から完璧を目指さない

次の章では、テストで防げない事象に向き合う技術 — **デバッグ・ログ・例外設計** に進みます。
