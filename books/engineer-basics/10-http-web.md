## この章のゴール

「Webサイト」「Webアプリ」「Web API」 — Web エンジニアの "Web" の中身は、ほぼすべて **HTTP** という1つのプロトコルで動いています。

この章では HTTP の **リクエスト/レスポンス、メソッド、ステータスコード、ヘッダー、Cookie、HTTPS** をひととおり押さえます。
これが分かるとブラウザの DevTools が急に読めるようになります。

## 10-1. HTTP とは

**HTTP (HyperText Transfer Protocol)** は、ブラウザとサーバーの間で文書をやりとりするためのプロトコルです。

特徴:
- **テキストベース** (HTTP/1.1 まで)
- **リクエスト/レスポンス** 形式
- **ステートレス** (1回のやりとりが独立。サーバーは前の通信を覚えていない)

## 10-2. リクエストとレスポンス

### リクエスト (クライアント → サーバー)

```http
GET /users/123 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJxxx...
```

構成:
1. **リクエスト行**: メソッド + パス + バージョン
2. **ヘッダー**: メタ情報 (送り元、認証、形式...)
3. **空行**
4. **ボディ** (POST/PUT のとき)

### レスポンス (サーバー → クライアント)

```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 56

{"id": 123, "name": "Hikaru", "email": "h@example.com"}
```

構成:
1. **ステータス行**: バージョン + コード + 説明
2. **ヘッダー**
3. **空行**
4. **ボディ** (HTML, JSON, 画像など)

## 10-3. HTTP メソッド

「サーバーに何をしたいか」を表す動詞。

| メソッド | 用途 | 冪等? | ボディ |
|---------|------|-------|-------|
| **GET** | 取得 | ◯ | なし |
| **POST** | 新規作成 / 副作用あり | × | あり |
| **PUT** | 全体更新 | ◯ | あり |
| **PATCH** | 部分更新 | △ | あり |
| **DELETE** | 削除 | ◯ | なし |
| HEAD | ヘッダーだけ取得 | ◯ | なし |
| OPTIONS | 利用可能なメソッド確認 | ◯ | なし |

**冪等 (idempotent)** = 何回呼んでも同じ結果。GET/PUT/DELETE は冪等、POST は非冪等。
これは第12章 API 設計で再登場します。

## 10-4. ステータスコード — 3桁の意味

レスポンスの最初に返る3桁の数字。**1桁目で大カテゴリが分かる** ようにできています。

| 範囲 | カテゴリ |
|------|---------|
| 1xx | 情報 (滅多に見ない) |
| 2xx | 成功 |
| 3xx | リダイレクト |
| 4xx | クライアントが悪い |
| 5xx | サーバーが悪い |

### 必ず覚えるべき10個

| コード | 意味 |
|-------|------|
| 200 | OK |
| 201 | Created (新規作成成功) |
| 204 | No Content (成功・ボディなし) |
| 301 | Moved Permanently (恒久リダイレクト) |
| 302 | Found (一時リダイレクト) |
| 304 | Not Modified (キャッシュ有効) |
| 400 | Bad Request (リクエストの形が悪い) |
| 401 | Unauthorized (未認証) |
| 403 | Forbidden (権限なし) |
| 404 | Not Found |
| 500 | Internal Server Error |
| 502 | Bad Gateway |
| 503 | Service Unavailable |

「401 と 403 の違い」「301 と 302 の違い」はよく面接で聞かれます。401 は **誰だか分からない**、403 は **誰か分かるけど許可なし**。301 は **永続的な移動 (SEOに反映)**、302 は **一時的**。

## 10-5. URL の構造

URL = **U**niform **R**esource **L**ocator。Webリソースの住所。

```
https://user:pass@api.example.com:443/users/123?lang=ja#profile
└─┬─┘   └──┬──┘  └──────┬──────┘ └┬┘ └──┬──┘ └─┬──┘ └──┬──┘
スキーム  認証情報  ホスト名         ポート パス  クエリ   フラグメント
```

| 部品 | 役割 |
|------|------|
| スキーム | http / https / ftp / ws ... |
| ホスト名 | サーバーのドメイン or IP |
| ポート | 省略時は http=80, https=443 |
| パス | サーバー内のリソース指定 |
| クエリ (`?key=val`) | 検索条件などの追加情報 |
| フラグメント (`#xxx`) | ページ内アンカー (サーバーには送られない) |

## 10-6. ヘッダー — 通信のメタ情報

リクエスト/レスポンスに付く「**追加情報**」。代表例:

### リクエストヘッダー

| ヘッダー | 用途 |
|---------|------|
| `Host` | どのドメイン宛か |
| `User-Agent` | クライアントの種類 |
| `Accept` | 受け取れる形式 |
| `Authorization` | 認証情報 |
| `Cookie` | サーバーが以前セットした値を返送 |
| `Content-Type` | ボディの形式 (送信時) |

### レスポンスヘッダー

| ヘッダー | 用途 |
|---------|------|
| `Content-Type` | ボディの形式 |
| `Content-Length` | ボディのバイト数 |
| `Set-Cookie` | クライアントに保存させたい値 |
| `Cache-Control` | キャッシュの扱い |
| `Location` | リダイレクト先 |

## 10-7. Cookie — ステートレスな世界に状態を持たせる仕組み

HTTP は本来ステートレス。でも「ログイン状態を保持したい」みたいなニーズが普通にあります。

これを実現するのが **Cookie**。

```
サーバー → クライアント:
  Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax

以後のリクエスト:
  Cookie: session_id=abc123
```

セキュリティ属性:
- **HttpOnly**: JavaScript から読めない (XSS対策)
- **Secure**: HTTPS のときだけ送る
- **SameSite**: 別サイトからの送信を制限 (CSRF対策)

第17章のセキュリティで詳しく扱います。

## 10-8. HTTPS — 暗号化と認証

HTTP に **TLS (旧 SSL)** という暗号化層を加えたのが HTTPS。
ブラウザの URL バーに「🔒」マークが出るやつです。

提供するもの:
- **暗号化**: 経路上で盗聴されても中身が読めない
- **改ざん検知**: 途中で書き換えられたら気づける
- **サーバー認証**: 接続先が本物のサーバーか確認 (証明書)

現代の Web では HTTPS が前提。Google Chrome は HTTP のサイトを「保護されていません」と表示します。
Let's Encrypt で証明書が無料で取れるようになり、個人サイトでも HTTPS が当然になりました。

## 10-9. HTTP のバージョン進化

| バージョン | 特徴 |
|----------|------|
| HTTP/1.0 | 1リクエストごとに接続を作り直す (遅い) |
| HTTP/1.1 | コネクション再利用 (Keep-Alive)、テキストベース |
| HTTP/2 | バイナリ、多重化、ヘッダー圧縮 |
| HTTP/3 | UDP ベース (QUIC)、接続確立が速い |

HTTP/3 はモバイル環境などで通信途切れに強い。徐々に普及中。

## 10-10. RESTful な使い方

リソースを URL で表現し、それに対する操作を HTTP メソッドで行うスタイル。

```
GET    /users        → ユーザー一覧
GET    /users/123    → ID=123 のユーザー
POST   /users        → 新規作成
PUT    /users/123    → 全体更新
DELETE /users/123    → 削除
```

詳細は第12章 API 設計で。

## 10-11. ブラウザの DevTools の Network タブを開こう

「習うより慣れろ」で、Chrome の DevTools の Network タブを開いてみてください。

任意のサイトを開くと:
- 何個のリクエストが飛んでいるか
- どれが何ミリ秒かかっているか
- どのステータスコードが返っているか
- ヘッダーは何が付いているか

がリアルタイムに見えます。HTTP の理解は **この画面を眺める時間** で深まります。

## まとめ

- HTTP は **リクエスト/レスポンス** 形式のテキストプロトコル (HTTP/1.1まで)
- **ステートレス** が原則、状態は Cookie で持たせる
- **メソッド** = やりたいこと: GET/POST/PUT/PATCH/DELETE
- **ステータスコード** の1桁目で大カテゴリ。最低10個は暗記
- **URL** = スキーム + ホスト + ポート + パス + クエリ
- **ヘッダー** で認証・形式・キャッシュなどメタ情報を伝える
- **Cookie** + HttpOnly/Secure/SameSite でステート管理
- **HTTPS = HTTP + TLS** で暗号化・改ざん検知・認証
- バージョン進化: 1.0 → 1.1 → 2 → 3 (QUIC/UDP)
