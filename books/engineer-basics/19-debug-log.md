## この章のゴール

新人とベテランの差が一番出るのが **デバッグ力** です。
バグが0のコードはありません。**「バグが出たとき、どれだけ早く原因にたどり着けるか」** が現場で生きる力です。

この章では、デバッグの考え方・印刷デバッグの限界とデバッガ・ログ設計・例外の扱いを扱います。

## 19-1. デバッグの基本姿勢

新人の多くが「**当てずっぽうで直そうとして時間を溶かす**」。これを防ぐのが第一歩。

正しい流れ:

1. **再現する** (同じ条件で必ず起きるか確認)
2. **仮説を立てる** (どこで何が起きているはず)
3. **検証する** (ログ・デバッガ・テストで仮説を確認)
4. **原因を特定する**
5. **直す**
6. **同じバグが二度と起きないようテストを足す**

「直す」は最後。**原因を確信してから手を動かす**。

## 19-2. 二分探索デバッグ

「どこが悪いか分からない」とき最強の手法。

- 直近の変更を半分まで戻す → 直る/直らない?
- それで切り分け、さらに半分に絞る

Git の `git bisect` はまさにこれを自動化したコマンドで、「どのコミットでバグが入ったか」を log n 回の試行で特定できます。

## 19-3. printデバッグの限界

最も気軽な手法。

```python
print(f"DEBUG: x = {x}")
```

長所:
- すぐ書ける
- 言語問わず効く

短所:
- 残ったまま本番に行く
- 大量に書くと逆に追えない
- 状態が変わるたびに編集が必要

「**短時間で済む調査は print、長くなりそうならデバッガ**」が現実的な使い分け。

## 19-4. デバッガの使い方

**ブレークポイント (一時停止) して、変数・呼び出し履歴を眺める** ツール。

主要IDE のデバッガ:
- VS Code (どの言語でも)
- PyCharm (Python)
- IntelliJ IDEA (Java/Kotlin)
- Chrome DevTools (JS)

最低限の操作:
- **ブレークポイント設定** (行の左クリック)
- **ステップ実行** (1行進む / 関数の中に入る / 関数を抜ける)
- **変数ウォッチ** (値の変化を追う)
- **コールスタック** (どこから来たか)

「ステップ実行で、自分の思っているのと違う変数の値が出たところがバグ」 — これがデバッガの基本テクニック。

## 19-5. ログとは何か

ログ = **動いているシステムが残す痕跡**。
本番では「ローカルで再現できない」が普通なので、ログだけが頼り。

### ログレベル

| レベル | 用途 |
|--------|------|
| **TRACE** | 超細かい開発時のみ |
| **DEBUG** | 詳細な動作、開発時に出す |
| **INFO** | 正常な動作 (ユーザー登録、ジョブ開始など) |
| **WARN** | 異常ではないが注意 (リトライ、デフォルト値使用) |
| **ERROR** | エラーだがシステムは続行 |
| **FATAL** | システム停止レベルの致命的問題 |

環境ごとに出すレベルを変える。本番は INFO 以上、開発は DEBUG 以上、など。

## 19-6. 構造化ログ (Structured Logging)

平文ではなく **JSON** など機械可読な形でログを出す。

❌ 平文:
```
2026-05-31 12:34:56 user 123 logged in from 192.168.1.10
```

✅ 構造化:
```json
{"timestamp": "2026-05-31T12:34:56Z", "level": "INFO", "event": "user_login", "user_id": 123, "ip": "192.168.1.10"}
```

後から「user_id=123 のログだけ抽出」「特定IPからの全イベント」のような検索・集計が即座にできる。
クラウド時代の前提です。

## 19-7. 良いログのための原則

1. **何が起きたか分かる**: 「Error occurred」だけは何も分からない。具体的に
2. **構造化する**: JSON で意味のあるキーで
3. **相関IDを付ける**: 1つのリクエストが複数サービスを通るとき、トレースできるID
4. **個人情報・秘密情報を出さない**: パスワード・トークン・メール本文などをログに残さない
5. **過剰に出さない**: 全リクエストにDEBUGログ100行とかやると本番で死ぬ
6. **時間・ホスト・サービス名を付ける**: マイクロサービスではどこのログか分からなくなる

## 19-8. 例外 (Exception)

エラーが起きたときに「処理の流れを **中断して上に投げ上げる**」仕組み。

```python
def divide(a, b):
    if b == 0:
        raise ValueError("division by zero")
    return a / b

try:
    result = divide(10, 0)
except ValueError as e:
    print(f"caught: {e}")
```

### try / except (catch) の使い方

| パターン | 推奨度 |
|---------|--------|
| 具体的な例外をキャッチして処理 | ◎ |
| 例外を握りつぶす (`except: pass`) | × NG |
| 例外を握りつぶしてログだけ | △ (場面次第) |
| 何でもキャッチ (`except Exception`) | △ 最後の砦のみ |

**握りつぶしは絶対NG**。バグの存在が見えなくなる最悪のパターン。

## 19-9. 例外設計の指針

### 1. 起こりうるか・例外的か

| | 通常制御フロー | 例外 |
|--|--------------|------|
| 例 | バリデーションエラー、見つからない | DB接続切断、メモリ不足 |
| 仕組み | 戻り値 / Result 型 | 例外 (throw) |

「**例外で制御フローを書かない**」というのが現代の流れ。バリデーション失敗は例外じゃなく Result/エラー値で返す方が、コードの意図が明確になる。

### 2. 例外は具体的に

```python
# ❌ 何でも RuntimeError
raise RuntimeError("error")

# ✅ 何が悪いか分かる
raise UserNotFoundError(f"user_id={user_id} not found")
```

カスタム例外クラスを作るのは推奨。後で `except UserNotFoundError:` で個別に扱える。

### 3. 起きた場所で握らず、責任者まで投げ上げる

低レイヤで起きた例外をその場で握りつぶすと、上位は「正常に動いた」と思い込んでバグの温床になる。

**「投げ上げて、責任者 (ハンドラ) のところでまとめて処理」** が定石。Web アプリなら HTTP エラーへの変換、CLI ならユーザーへのエラー表示まで持っていく。

### 4. リソースを必ず解放する

```python
# ❌
f = open("file.txt")
data = f.read()  # ここで例外が起きるとファイルが閉じない
f.close()

# ✅
with open("file.txt") as f:
    data = f.read()  # 例外が起きても必ず close される
```

`try/finally`、Python の `with`、Java の `try-with-resources`、Go の `defer` — どの言語にも仕組みがあります。

## 19-10. スタックトレースの読み方

例外が起きたときに出る、関数呼び出しの履歴。

```
Traceback (most recent call last):
  File "app.py", line 42, in main
    process_user(user_id)
  File "app.py", line 30, in process_user
    user = fetch_user(user_id)
  File "app.py", line 18, in fetch_user
    return db.query(User).get(user_id)
ValueError: user_id must be positive
```

**読む順番は「下から」** — 一番下が **直接の原因**、上に行くほど **呼び出し元**。
スタックトレースをちゃんと読めるだけで、調査時間が一気に減ります。

## 19-11. 分散トレーシング (Distributed Tracing)

マイクロサービス時代、1つのリクエストが10個のサービスを通ることがあります。
**「どのサービスで何ミリ秒かかったか」を可視化** するのが分散トレーシング。

ツール:
- Jaeger
- Zipkin
- OpenTelemetry (= 標準仕様、各ツールに送れる)
- AWS X-Ray, Datadog APM

リクエストごとに **trace_id** を付け、各サービスのログ・スパンに伝播させる。これで「どこが遅いか」が一目で分かる。

## 19-12. 観測可能性 (Observability) 3点セット

現代の運用は次の3つを統合管理します。

- **メトリクス**: CPU使用率、リクエスト数、エラー率 (Prometheus, Datadog)
- **ログ**: 文字列イベント (CloudWatch Logs, Loki, Datadog Logs)
- **トレース**: リクエストの経路 (Jaeger, AWS X-Ray)

詳しくは第23章で扱います。

## まとめ

- デバッグは **再現 → 仮説 → 検証 → 特定 → 修正 → テスト追加**
- 「どこが悪い分からない」は **二分探索** で潰す (`git bisect` も同じ)
- 軽い調査は print、本気のときは **デバッガ** + ブレークポイント + ステップ実行
- ログは **構造化 (JSON) + 適切なレベル + 相関ID** がベスト
- ログに **秘密情報を出さない**、過剰に出さない
- 例外を **握りつぶさない**、具体的な例外クラスを使う
- バリデーションは例外より **Result 値** で返すと意図が明確
- リソース解放は **with / try-finally / defer**
- スタックトレースは **下から読む**
- マイクロサービス時代は **分散トレーシング + メトリクス + ログ** の Observability 3点セット
