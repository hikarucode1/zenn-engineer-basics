title: "NoSQL とキャッシュ"

## この章のゴール

RDB は強力ですが、すべての問題に最適とは限りません。

「億単位のドキュメントを保存したい」「ミリ秒単位の応答が要る」「データ構造が頻繁に変わる」 — このような場面で登場するのが **NoSQL** と **キャッシュ** です。

この章では4種類の NoSQL と、その代表格 Redis / MongoDB / DynamoDB / Elasticsearch、そして CAP 定理を扱います。

## 15-1. NoSQL とは

**NoSQL = "Not Only SQL"**。RDB ではない (あるいは RDB と共存する) DB の総称。

特徴 (RDB との対比):

| | RDB | NoSQL |
|--|-----|-------|
| スキーマ | 固定 | 柔軟・なし |
| トランザクション | 強い (ACID) | 弱い (BASE) |
| スケール | 垂直 (1台を強く) | 水平 (台数で) |
| クエリ | SQL で何でも | 単純・限定的 |
| 速度 | 普通 | 用途次第で爆速 |

「**RDB の代わり**」ではなく、「**RDB の弱点を補う特化型**」と捉えるのが正しい姿勢です。

## 15-2. NoSQL の4分類

| 種類 | 代表 | データ形式 |
|------|------|----------|
| **キーバリュー** | Redis, DynamoDB | キー → 値 |
| **ドキュメント** | MongoDB, Firestore | キー → JSON 風文書 |
| **ワイドカラム** | Cassandra, ScyllaDB | 列指向の巨大表 |
| **グラフ** | Neo4j | ノードと辺 |

## 15-3. キーバリューストア (KVS)

最もシンプルな DB。**キー → 値** の辞書をネットワーク越しに引くだけ。

```
SET user:123 "{name: Hikaru, age: 25}"
GET user:123  → "{name: Hikaru, age: 25}"
```

### Redis

メモリ上で動く KVS の代表格。**マイクロ秒応答** の速さ。

主な用途:
- **セッション保存** (ログイン情報)
- **キャッシュ** (DB の結果を一時保存)
- **レートリミッタ** (カウンタ)
- **タスクキュー** (ジョブ管理)
- **リアルタイムランキング** (ソート済みセット)

機能が単純な KVS にとどまらず、**リスト・ハッシュ・セット・ソート済みセット・ストリーム** などのデータ構造が標準で用意されています。

### DynamoDB

AWS が提供するマネージド KVS / ドキュメントの中間。
**任意のスケール** でほぼ常に1桁ミリ秒で応答するのが売り。グローバルなサービスで使われる。

## 15-4. ドキュメントストア

**JSON ライクな構造体** をそのまま保存できる DB。

```json
{
  "_id": "user_123",
  "name": "Hikaru",
  "tags": ["python", "iot"],
  "address": { "city": "Tokyo", "zip": "100-0001" }
}
```

スキーマがなく、追加フィールドを自由に入れられる。

### MongoDB

最も普及したドキュメント DB。
- **試作・初期開発** に向いている (スキーマ変更が楽)
- 集計パイプライン、地理空間クエリなど機能豊富
- 大規模では設計が難しく、無策で使うと痛い目に

### Firebase Firestore

Google 製、モバイル/Webアプリ向け。**リアルタイム同期** が標準でついてくる。

## 15-5. ワイドカラム — Cassandra / ScyllaDB

「行ごとに違う列を持てる、巨大な分散表」のような構造。
書き込みが滅茶苦茶速く、PB (ペタバイト) クラスのデータを扱える。

用途:
- 時系列データ (センサー、ログ)
- メッセージ履歴 (Netflix, Discord 等で採用例)

学習コストは高い。**実務でいきなり選ぶことは少ない** ですが、概念は知っておきましょう。

## 15-6. グラフDB — Neo4j

ノードと辺を直接データ構造として扱う。

例: 「Aさんと友達の友達のうち、東京に住んでいる人」のような関係をたどるクエリが超高速。

用途:
- SNS のフレンド推薦
- 不正検知 (取引関係のたどり方)
- ナレッジグラフ

## 15-7. 検索エンジン系 — Elasticsearch / OpenSearch

「**全文検索**」に特化した DB。
日本語のあいまい検索、形態素解析、スコアリング、ファセット (絞り込み) — Web の検索画面の裏でほぼ必ず使われています。

```
PUT /products/_doc/1
{"name": "ノートパソコン", "tags": ["pc", "mobile"]}

GET /products/_search
{"query": {"match": {"name": "ノート"}}}
```

「DBの WHERE LIKE で頑張る」のをやめて Elasticsearch に逃がすのが定石。

## 15-8. CAP 定理 — 分散DBのジレンマ

分散システム理論の根本。**C、A、P の3つは同時に満たせない**、というもの。

- **C**onsistency (一貫性): どのノードに聞いても最新の同じデータ
- **A**vailability (可用性): いつでも応答が返ってくる
- **P**artition tolerance (分割耐性): ネットワーク分断が起きても動く

現実のネットワークは必ず P (障害) が起きるので、**C か A のどちらを取るか** の選択になります。

| 指向 | DB 例 |
|------|------|
| CP (一貫性優先) | MongoDB (古典設定), HBase |
| AP (可用性優先) | Cassandra, DynamoDB, Riak |

RDB は単一ノード前提で ACID を強く保証 (= C 寄り) ですが、分散すると CP/AP のトレードオフに直面します。

## 15-9. BASE — NoSQL 側の哲学

ACID に対比される NoSQL の設計指針が **BASE**。

| 文字 | 意味 |
|------|------|
| **B**asically **A**vailable | だいたい動く |
| **S**oft state | 状態は時間で変化する |
| **E**ventual consistency | 最終的には一貫する |

「すぐ正しい」ではなく「**いつか正しくなる**」を許容することで、可用性とスケーラビリティを取る考え方。

## 15-10. キャッシュ — 速い場所にコピーを置く

キャッシュは DB そのものというより **戦略**。
「DBの結果をメモリに一時保存して、次回からそこから返す」だけ。

```
最初のアクセス: Client → API → DB (遅い) → Cache に保存 → Client
2回目以降:    Client → API → Cache (速い) → Client
```

代表的なキャッシュ層:

| 層 | 例 |
|----|----|
| CDN | Cloudflare, CloudFront (静的ファイル) |
| アプリ手前 | Varnish, Nginx |
| アプリ内 | Redis, Memcached |
| DB 内 | RDB のクエリキャッシュ |
| ブラウザ | HTTP キャッシュ, Service Worker |

### キャッシュの2大問題

1. **キャッシュ無効化 (invalidation)**: 元データが変わったらキャッシュも更新/削除しないと古いデータを返してしまう
2. **命名 (naming)**: キャッシュキーの設計

**「コンピュータサイエンスの2大難問はキャッシュの無効化と命名だ」** — フィル・カールトン

### キャッシュ戦略

| 戦略 | 動き |
|------|------|
| **Cache-Aside** | アプリがDBとキャッシュを両方触る (最も一般的) |
| **Read-Through** | キャッシュ層がDBを背負う |
| **Write-Through** | 書き込み時にキャッシュも更新 |
| **Write-Behind** | キャッシュにだけ書いて非同期にDBへ |

### TTL (Time To Live)

「**何秒で自動的に消えるか**」をキャッシュに設定する仕組み。
TTL を入れずに永続キャッシュにすると、必ず古いデータ問題が起きます。

## 15-11. ポリグロット・パーシステンス

「**1つのアプリで複数のDBを併用する**」設計。現代では普通の選択肢です。

例: メッセージアプリ
- **PostgreSQL**: ユーザー情報、認証 (ACIDが必要)
- **Redis**: セッション、最新タイムライン (高速)
- **Elasticsearch**: 全文検索
- **S3**: 添付画像/動画
- **DynamoDB**: メッセージ本体 (スケール)

「最強の DB」を1つ選ぶより、「**それぞれの強みを束ねる**」のがモダンな選択。

## 15-12. 「で、新人は何を使えばいい?」

迷ったら次の順で考えてください。

1. まず **PostgreSQL** で組む
2. セッションとレートリミット用に **Redis** を足す
3. 検索が遅くなってきたら **Elasticsearch** を考える
4. 1億件超え/書き込み激重になったら他の選択肢

つまり「PostgreSQL + Redis」 が現代のスタートラインで、これで多くのサービスは十分です。

## まとめ

- NoSQL = RDB の代替ではなく **弱点を補う特化型 DB** の総称
- **KVS / ドキュメント / ワイドカラム / グラフ** の4分類
- **Redis** = メモリ上の KVS、キャッシュ・セッション・カウンタの定番
- **MongoDB** = JSON 文書 DB、初期開発が早い
- **Elasticsearch** = 全文検索の決定版
- **CAP 定理**: 一貫性・可用性・分割耐性のうち同時に満たせるのは2つ
- **ACID vs BASE**: 強い整合性 vs いつか整合する
- **キャッシュ** は戦略。Cache-Aside が最も使う。TTL 必須
- 現代は **ポリグロット・パーシステンス** (複数 DB 併用) が普通
- 学生・新人は **PostgreSQL + Redis** から始めればOK

次の章では、これらをどう組み合わせるか — **データモデリング** の考え方に進みます。
