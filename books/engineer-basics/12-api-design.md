## この章のゴール

フロントとバックエンドが会話するときに使う「**取り決め**」が API です。

ここでは Web 開発で最も使われる **REST API**、その対抗として登場した **GraphQL**、そして両者の使い分け・設計の作法を学びます。
独学で API を作ると「動くけど何かおかしい」設計になりがちなので、**何が良い API か** の感覚をここで身につけましょう。

## 12-1. API とは

**API (Application Programming Interface)** = アプリ同士のやりとりの窓口。
Web API はその中で **HTTP 経由でやりとりする API** のこと。

```
[フロント] --HTTP--> [API] --DB--> [データ]
```

API 設計が良いと:
- 別チームが理解しやすい
- フロントが楽になる
- 将来の変更に強くなる

逆に悪いと、改修のたびに連絡会議が必要になります。

## 12-2. REST API の基本

**REST (Representational State Transfer)** は2000年頃に提唱された設計スタイル。
HTTP の仕組みを素直に使い、**リソースを URL で表現する**。

### 基本ルール

1. **リソース** を URL で表現する (名詞、複数形が定番)
2. **動詞は HTTP メソッド** で表す
3. **ステータスコード** で結果を伝える
4. **JSON** でデータをやりとりする (現代の標準)

### 典型例

```
GET    /users          → ユーザー一覧
GET    /users/123      → ID=123 を取得
POST   /users          → 新規作成
PUT    /users/123      → 全体更新
PATCH  /users/123      → 部分更新
DELETE /users/123      → 削除
```

### ネストでリソースを表現

```
GET    /users/123/posts        → ユーザー123の投稿一覧
GET    /users/123/posts/45     → ユーザー123の投稿ID 45
```

ただしネストは2階層までが推奨。深くなりすぎると読めなくなる。

## 12-3. REST 設計のよくあるアンチパターン

新人がやりがちな失敗例。

| ダメな例 | なぜダメか |
|---------|-----------|
| `GET /getUsers` | 動詞を URL に入れている (メソッドが GET なら不要) |
| `POST /users/delete/123` | DELETE を使うべき |
| `GET /users?action=delete` | クエリで動詞は危険 |
| `GET /api/v1/user/list/all` | 階層が冗長 |
| `200 OK + {"error": "Not found"}` | エラーは 404 で返す |

REST の真髄は「**動詞はメソッド、名詞は URL**」。これだけ守れば8割正しいです。

## 12-4. ステータスコードの使い分け

API でよく返すコード:

| ケース | コード |
|-------|-------|
| 取得成功 | 200 OK |
| 作成成功 | 201 Created |
| 成功・返すものなし (DELETE等) | 204 No Content |
| パラメータが不正 | 400 Bad Request |
| 認証されていない | 401 Unauthorized |
| 認証OKだが権限なし | 403 Forbidden |
| 存在しない | 404 Not Found |
| メソッドが許可されていない | 405 Method Not Allowed |
| 衝突 (重複作成など) | 409 Conflict |
| バリデーションエラー | 422 Unprocessable Entity |
| 過剰アクセス | 429 Too Many Requests |
| サーバー側のバグ | 500 Internal Server Error |

「全部 200 + body にエラー」の API は、最初は楽でも後で必ず後悔します。

## 12-5. ページネーション

「100万件のレコードを全部一度に返す」のはダメ。区切って返します。

### オフセット方式

```
GET /users?page=2&per_page=20
```

実装が簡単。ただしデータが入れ替わると重複や漏れが起きる。

### カーソル方式

```
GET /users?after=xxxxx&limit=20
```

「次へ」のリンクをサーバーが渡す。安定して取得できるが実装はやや複雑。Twitter API などが採用。

## 12-6. 認証と認可 — API Key, Bearer Token, OAuth

リクエストの送り主が誰かを示す仕組み。

### Bearer Token (JWT が代表)

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

「**この文字列を持っていることが認証**」というシンプルな仕組み。
JWT (JSON Web Token) は内部に「ユーザーID」「有効期限」などを含み、サーバーが署名する。

### API Key

```
X-API-Key: abcd1234
```

「外部の開発者向けに公開する API」でよく使う。
Stripe、SendGrid のような SaaS の API はだいたいこれ。

### OAuth 2.0

第三者アプリに「ユーザーのデータを許可した範囲だけ触らせる」仕組み。
「Google でログイン」「GitHub で連携」はこれ。

詳細は第17章のセキュリティで再登場します。

## 12-7. レート制限 (Rate Limiting)

1ユーザーから秒間1000回のリクエストが来たら、サーバーが死にます。これを防ぐのが **レート制限**。

```http
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
Retry-After: 60
```

「1分あたり100リクエストまで」「超えたら 429 を返す」のが定番。
クライアント側はこれを見て **指数バックオフ** で再試行するのが行儀の良い実装。

## 12-8. バージョニング

API は一度公開すると簡単に変えられません。**バージョンを明示** することで後方互換を保ちます。

| 方式 | 例 |
|------|----|
| URL に入れる | `/api/v1/users` |
| ヘッダーに入れる | `Accept: application/vnd.myapi.v2+json` |
| クエリで指定 | `/users?v=2` |

最も多いのは URL 方式。「**破壊的変更が出たら v2 を切る**」のが定石です。

## 12-9. GraphQL — クライアントが欲しい形を指定する

REST の対抗として Facebook が公開したスタイル。

### REST の悩み

- 「ユーザー情報と投稿一覧をまとめて欲しい」 → 2回 API を呼ぶか、専用エンドポイントを作るしかない
- 「ユーザーの name と email だけ欲しい」 → 余計なフィールドも返ってくる

### GraphQL の解

**1つのエンドポイントに、欲しい形をクエリで送る**。

```graphql
query {
  user(id: 123) {
    name
    email
    posts {
      title
    }
  }
}
```

レスポンス:
```json
{
  "user": {
    "name": "Hikaru",
    "email": "h@example.com",
    "posts": [{ "title": "Hello GraphQL" }]
  }
}
```

利点:
- 必要なデータだけ取れる (オーバーフェッチ解消)
- 1リクエストで関連データをまとめて取れる
- 型 (スキーマ) がきっちり定義されている

欠点:
- キャッシュが難しい (URL ベースじゃない)
- N+1 問題が起きやすい
- 学習コストが高め

「**REST が辛い大規模アプリで GraphQL に乗り換える**」が現実的な選択。

## 12-10. gRPC — マイクロサービス間で速さ重視

Google が公開した RPC フレームワーク。
HTTP/2 上でバイナリ (Protocol Buffers) を流す。

- 型がきっちり、コード生成あり
- バイナリで軽量・高速
- ブラウザから直接叩きにくい (プロキシ要)

**サーバー間通信 (マイクロサービス内部)** でよく採用されます。

## 12-11. WebSocket — 双方向リアルタイム

通常の HTTP は「クライアントが聞いて、サーバーが答える」一方向。
**WebSocket** は接続を張りっぱなしにして、サーバーからもプッシュできる仕組み。

用途:
- チャット
- 通知
- 同時編集 (Figma, Notion など)
- 株価のリアルタイム配信

## 12-12. API ドキュメントとスキーマファースト

API は **触る人が読めるドキュメント** がないと使われません。

定番ツール:
- **OpenAPI (旧 Swagger)**: REST のスキーマ + ドキュメント生成
- **GraphQL Schema**: GraphQL は標準でスキーマがある
- **Postman / Insomnia**: API を叩いて試すクライアント

**スキーマファースト** = 実装の前にスキーマを書き、それを元にコード・ドキュメント・テストを生成するアプローチ。チーム開発で強力。

## まとめ

- API = アプリ同士のやりとりの窓口、Web API は HTTP ベース
- **REST**: リソースを URL で表現、動詞は HTTP メソッド
- ステータスコードを正しく使う (200で全部済ませない)
- **ページネーション** はオフセット式かカーソル式
- 認証は **Bearer Token (JWT)**、外部公開は **API Key**、連携は **OAuth**
- **レート制限** で過剰アクセスから守る、429 を返す
- **バージョニング** で後方互換を保つ
- **GraphQL** はクライアントが形を指定できる。大規模で有効
- **gRPC** はサーバー間通信、**WebSocket** はリアルタイム
- **OpenAPI / GraphQL Schema** で **スキーマファースト** な開発を
