## この章のゴール

普段あなたが書く `print("Hello")` の一行が、なぜ画面に文字を映し出せるのか。
その「魔法」を解くには、まずコンピュータという機械の中身を眺める必要があります。

この章では、**CPU・メモリ・ストレージ・I/O** という4つの登場人物が、どんな役割分担で協力しているのかを俯瞰します。細部はこの先の章で深掘りするので、ここでは「だいたいこんな関係」を掴むことを目指してください。

:::message
新人エンジニアが最初につまずきがちなのは、「ソフトウェアの話なのに、なぜハードの話を知らないといけないのか?」という疑問です。答えは単純で、**ソフトウェアの性能・挙動・限界はすべてハードウェアに由来する**からです。OOMで落ちるのも、ファイルI/Oが遅いのも、根っこは物理です。
:::

## 1-1. コンピュータを構成する4つの登場人物

ものすごく雑に言うと、コンピュータは次の4つでできています。

| 役割 | 担当 | 例えるなら |
|------|------|----------|
| 計算する | **CPU** (中央演算装置) | 料理人 |
| 一時的に置く | **メモリ** (RAM、主記憶) | 調理台 |
| 永続的に保管する | **ストレージ** (SSD/HDD、補助記憶) | 冷蔵庫・倉庫 |
| 外と話す | **I/O** (キーボード・画面・ネット) | 客との接客窓口 |

ある朝のレストランを想像してみてください。

1. 客 (=ユーザー) が「ハンバーグください」と注文する → **入力 (Input)**
2. 料理人 (=CPU) が冷蔵庫 (=ストレージ) から肉を取り出す
3. 取り出した肉を調理台 (=メモリ) に並べて、フライパンで焼く
4. 焼き上がったハンバーグを客に出す → **出力 (Output)**

このフローはあなたが書くプログラムでも完全に同じです。
ファイルを開き → メモリに読み込み → CPUが処理し → 画面やファイルに出力する。アプリの種類が変わっても、この骨格は変わりません。

## 1-2. CPU — 命令を1つずつ実行する単純な機械

CPU の仕事は驚くほど単純です。延々と次のことを繰り返しているだけです。

1. **Fetch (取ってくる)**: メモリから命令を1つ読み出す
2. **Decode (読み解く)**: それがどんな命令か判別する
3. **Execute (実行する)**: 計算する/値を移動する/比較する
4. 次の命令へ進む

このループを **命令サイクル** と呼びます。
現代の CPU は1秒間にこのループを数十億回回します (3GHz なら毎秒30億回)。あなたの目には一瞬で動いたように見えるアプリも、内側ではこの単純作業の積み重ねです。

### レジスタ — CPUの中の「手元」

CPU の中には **レジスタ** と呼ばれる、ごく少数の超高速な記憶領域があります。
料理人が両手で持てる食材の数に限りがあるのと同じで、CPU は一度にレジスタの数だけしか値を扱えません。それ以外の値は全部メモリに置いておきます。

「CPU の中だけで完結する処理は速く、メモリにアクセスし始めると遅くなる」という直感は、後で出てくる **キャッシュ** や **計算量** の話を理解する土台になります。

## 1-3. メモリ — 速いが消える作業台

メモリ (RAM) は CPU のすぐ隣にある作業台です。

| | メモリ (RAM) | ストレージ (SSD) |
|--|-------------|----------------|
| 速度 | ◎ ナノ秒オーダー | ○ マイクロ〜ミリ秒 |
| 容量 | 数GB〜数十GB | 数百GB〜数TB |
| 電源切ると | **消える** | 残る |
| 価格/GB | 高い | 安い |

メモリは「速いけど切ったら消える」「容量に限りがある」という性質を持ちます。
プログラムが起動している間だけ、変数やオブジェクトはメモリ上に存在します。プログラムを終了すれば全部消えます。だから永続化したいものはファイルやDBに書き出す必要がある — これがあとで出てくる **永続化 (Persistence)** の話の出発点です。

:::message
「メモリリーク」「OutOfMemoryError」「ヒープ」「スタック」といった単語は、すべてこの "限りある作業台" をどう使うかの話です。第3章 (OS) と第6章 (データ構造) でもう一度出てきます。
:::

## 1-4. ストレージ — 遅いが消えない倉庫

ストレージは電源を切っても消えない記憶領域です。SSD・HDD・USBメモリなどがこれにあたります。

CPU はストレージから直接命令を実行できません。**必ず一度メモリに読み込んでから** 処理します。
これは料理人が冷蔵庫の中で調理しないのと同じで、作業効率の問題です。冷蔵庫(ストレージ)はメモリの数百〜数万倍遅いので、頻繁にアクセスするとプログラム全体が遅くなります。

「ファイル読み込みが遅い」「DB クエリが遅い」と感じる多くのケースで、根本原因はこの **メモリとストレージの速度差** です。

## 1-5. I/O — 外の世界とのインターフェース

I/O (Input/Output) は、コンピュータと外の世界をつなぐ入出力です。

- **入力**: キーボード、マウス、マイク、カメラ、センサー、ネットワークから届くパケット
- **出力**: ディスプレイ、スピーカー、プリンタ、ネットワークへ送るパケット

ネットワーク (Wi-Fi/Ethernet) も I/O の一種です。「Web アプリ」とは、突き詰めれば *ネットワーク I/O で受け取ったリクエストを CPU で処理し、ネットワーク I/O で返す* プログラムのことです。

I/O は CPU やメモリと比べて圧倒的に遅いので、現代のプログラミングでは **「I/O を待っている間に別のことをする」** ための工夫 (非同期、並行処理) が大きなテーマになります。これは第3章と第10章で改めて触れます。

## 1-6. バス — 登場人物をつなぐ配線

CPU・メモリ・I/O は **バス** と呼ばれる配線で結ばれています。
バスは「データの通り道」で、太い (=帯域が広い) ほど一度に多くのデータを運べます。

ここで覚えてほしいのはバスの構造そのものより、**「速度の階層」** があるという事実です。

```
速い ←──────────────────────────────────────→ 遅い
CPUレジスタ → CPUキャッシュ → メモリ → SSD → HDD → ネットワーク
```

数字で見るとこんなオーダー感です (機種により変動)。

| 場所 | 1回のアクセスにかかる時間 | レストラン例え |
|------|-----------------------|----------------|
| レジスタ | < 1 ns | 手の中 |
| L1 キャッシュ | 〜1 ns | 調理台のすぐ前 |
| メモリ | 〜100 ns | 厨房の棚 |
| SSD | 〜100 μs (= 10万 ns) | 隣の部屋の冷蔵庫 |
| HDD | 〜10 ms (= 1000万 ns) | 地下の倉庫 |
| ネットワーク (国内) | 〜10 ms | 別の店に電話 |

レジスタとメモリで100倍、メモリとSSDでさらに1000倍。**桁が一つ違うと体感が変わる**、というのがこの表の本質です。「なんとなく遅い」と感じたら、まずデータがどの階層を行き来しているかを疑うのが定石です。

## 1-7. プログラムが動くまでの全体図

最後に、あなたが書いた一行のプログラムが動くまでの流れを並べてみます。

1. ソースコードをコンパイラ/インタプリタが **機械語の命令列** に変換する
2. OS がその命令列を **ストレージ → メモリ** に読み込む (これを「プログラムをロードする」と言います)
3. OS が CPU に対して「ここから実行しろ」と指示する
4. CPU が **Fetch → Decode → Execute** をひたすら繰り返す
5. 必要に応じて I/O (画面・ファイル・ネット) とやりとりする
6. 処理が終わったら、メモリ上の領域を OS に返して終了する

このうち「OS が〜」の部分は第3章で、「コンパイラが〜」の部分は第5章で詳しく扱います。

## まとめ

- コンピュータは **CPU・メモリ・ストレージ・I/O** の4つの登場人物で動いている
- CPU は **Fetch → Decode → Execute** を繰り返すだけの単純な機械で、その単純さが秒間数十億回の繰り返しを支えている
- メモリは速いが切れると消える / ストレージは遅いが残る / I/O はさらに遅い
- 速度には **階層 (レジスタ → キャッシュ → メモリ → SSD → ネット)** があり、桁が変わると体感が変わる
- 「ソフトが遅い」「メモリが足りない」といった現場の問題は、すべてこの物理的な制約に由来する

次の章では、コンピュータが扱う「データ」そのものに目を向けます。**なぜすべてが 0 と 1 で表現できるのか**、その正体に迫ります。
