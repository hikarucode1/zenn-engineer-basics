## この章のゴール

ここから第2部「プログラミングの基礎」に入ります。

「プログラミング言語」と一口に言っても、Python と C と JavaScript と Go では別の生き物です。
この章では、**そもそもプログラミング言語って何者?** **なぜ何種類もあるの?** **コンパイルとインタプリタって何が違うの?** という根本に踏み込みます。

## 5-1. プログラミング = 機械への翻訳

CPU が直接読めるのは、第1章で見た **機械語 (= 0/1 の並び)** だけです。

```
01000001 00100000 00000001
```

これを人間が直接書くのは無理。なので **「人間に読みやすい記号」 → 「機械語」 への翻訳ルール** を作りました。これが **プログラミング言語** です。

翻訳の歴史:

| 世代 | 例 | 特徴 |
|------|----|----|
| 機械語 | `01000001...` | CPUが直接読む |
| アセンブリ | `MOV AX, 1` | 機械語の命令に名前をつけただけ |
| 高級言語 | `x = 1` | 人間が読める。コンパイラ/インタプリタが翻訳 |

現代のあなたが触る言語 (Python, Java, JS, Go, Rust, Swift...) はすべて **高級言語** です。

## 5-2. コンパイラ型 と インタプリタ型

高級言語を機械語にする方法は大きく2つに分かれます。

### コンパイラ型 (例: C, Go, Rust)

**実行前に丸ごと機械語に変換** してから動かす方式。

```
ソースコード (.c)
   ↓ コンパイル
実行ファイル (.exe / バイナリ)
   ↓ 実行
動く
```

特徴:
- 実行は速い (= 既に機械語になっている)
- 配布されるのは実行ファイル (ソースは隠せる)
- ビルドに時間がかかる
- OS/CPU ごとにビルドし直す必要がある

### インタプリタ型 (例: Python, Ruby, JavaScript)

**1行ずつ読みながら実行** する方式。

```
ソースコード (.py)
   ↓ インタプリタが1行ずつ読む
動く
```

特徴:
- すぐ書いてすぐ動く (試行錯誤しやすい)
- 実行は遅い (毎回翻訳)
- ソースをそのまま配布
- OS が違ってもインタプリタさえあれば動く

### 中間 (バイトコード) 型 (例: Java, C#, Python 内部)

**バイトコード** という中間形式に一度変換し、**仮想マシン (VM)** がそれを実行する方式。

```
Java
   ↓ コンパイル
.class (バイトコード)
   ↓ JVM が実行
動く
```

「コンパイラの速さ」と「インタプリタの移植性」のいいとこ取りを狙った設計。Python も実は内部で `.pyc` バイトコードを作っています。

## 5-3. なぜ言語は何種類もあるのか

「最強の言語を一つ作って終わりにしよう」とならない理由は、**目的・性能・書きやすさ・コミュニティのトレードオフ** があるからです。

| 言語 | 強みの領域 | 代償 |
|------|-----------|------|
| C / C++ | OS、ゲームエンジン、組み込み | 手動メモリ管理が大変 |
| Rust | 高速 + 安全 | 学習コスト高 |
| Go | サーバー、並行処理 | 表現力をあえて削っている |
| Java / Kotlin | 業務アプリ、Android | 起動が遅め |
| Swift | iOS / macOS | Apple 圏外で弱い |
| Python | データ分析、AI、スクリプト | 実行速度が遅い |
| Ruby | Web アプリ (Rails) | エコシステム縮小傾向 |
| JavaScript / TypeScript | ブラウザ、Node.js | 言語仕様の歴史的負債 |

「どれが正解」ではなく「**目的に合うものを選ぶ**」が答え。学生のうちは Python + JavaScript の2つを押さえれば領域はだいたい広く触れます。

## 5-4. 言語の共通要素 — 何を学んでも同じこと

新しい言語に触れるたびゼロから学ぶように見えますが、実は **どの言語にもほぼ同じ概念** があります。

| 要素 | 説明 |
|------|------|
| **変数** | 値に名前を付ける箱 |
| **型** | 値の種類 (整数・文字列・真偽値…) |
| **演算子** | `+ - * /` や `== !=` |
| **制御構造** | `if`, `for`, `while` |
| **関数** | 処理に名前を付けて呼び出せる単位 |
| **配列/オブジェクト** | データのまとまり |
| **エラー処理** | `try / catch` 系 |
| **モジュール / import** | 他のファイルを読み込む |

一つの言語を腰を据えて学ぶと、2つ目以降は「文法が違うだけ」で1〜2週間で書けるようになります。最初の言語をどれだけ深く理解するかが、その後の学習速度を決めます。

## 5-5. 静的型付け と 動的型付け

変数の型をどう扱うかで言語は大きく分かれます。

### 静的型付け (C, Java, Go, Rust, TypeScript)

変数の型を**書く側が明示**し、コンパイル時にチェック。

```typescript
let age: number = 25;
age = "hello"; // ← コンパイルエラー (実行する前に気づける)
```

### 動的型付け (Python, JavaScript, Ruby)

型を書かない。実行時に決まる。

```python
age = 25
age = "hello"  # OK (実行中に型が変わる)
```

| | 静的 | 動的 |
|--|------|------|
| 早く書き始める | △ | ◎ |
| 大規模になった時 | ◎ (型でバグを早期発見) | △ |
| 実行速度 | ◎ | △ |
| 学習しやすさ | △ | ◎ |

**業界トレンドは「動的言語に型を後付けする」** 方向です (例: Python の型ヒント、JS → TypeScript)。

## 5-6. プログラムが動くまで

第3章の OS の話とつなげると、こうなります。

1. あなたがソースコード `hello.py` を書く
2. **インタプリタ or コンパイラ** が中間形式 (バイトコード) or 機械語に翻訳
3. **OS** がそれをメモリに読み込み、CPU に実行を依頼
4. **CPU** が Fetch → Decode → Execute を繰り返す
5. 必要に応じてシステムコールで I/O (画面・ファイル・ネット) と通信

つまり「動くプログラム」は **言語処理系・OS・ハードウェアの3層** が連携した結果です。

## 5-7. パッケージマネージャ と エコシステム

現代のプログラミング言語にはほぼ必ず **パッケージマネージャ** が付属します。

| 言語 | パッケージマネージャ |
|------|--------------------|
| Python | pip, poetry, uv |
| JavaScript / Node | npm, yarn, pnpm |
| Ruby | gem, bundler |
| Java | maven, gradle |
| Rust | cargo |
| Go | go mod |

パッケージマネージャの役割:
- 他人が作ったライブラリ (= パッケージ) のインストール
- バージョン管理
- 依存関係の解決

「言語そのもの」より「**その言語のエコシステム (ライブラリの豊富さ)**」が選定理由になることも多いです。Python が AI で強いのは NumPy / PyTorch / Transformers があるから、JavaScript が Web で強いのは React / Next.js があるから。

## まとめ

- プログラミング言語 = **人間が書ける記号 → 機械語** への翻訳ルール
- **コンパイラ型 / インタプリタ型 / 中間バイトコード型** の3方式がある
- 言語が何種類もあるのは目的別のトレードオフがあるため
- 言語が違っても **変数・型・制御構造・関数** は共通
- **静的型 / 動的型** のトレードオフ。トレンドは動的言語への型の後付け
- 動くプログラム = **言語処理系 + OS + ハードウェア** の3層連携
- **パッケージマネージャ** とエコシステムが言語選定の半分を決める
