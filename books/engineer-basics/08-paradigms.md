## この章のゴール

「同じ機能」を作るのに、書き方は何通りもあります。

その「**書き方の流派**」を **プログラミングパラダイム** と呼びます。
本章では、現代の主要な3つ — **手続き型・オブジェクト指向 (OOP)・関数型** — の発想と使い分けを学びます。

「どれが正解」ではなく、「**問題に合わせて使い分ける**」のが現代的なスタイルです。

## 8-1. 手続き型 (Procedural)

最も素朴な書き方。**「上から順に処理を並べる」**。

```python
def calculate_total(items):
    total = 0
    for item in items:
        total += item["price"] * item["quantity"]
    if total > 10000:
        total *= 0.9  # 1万円超で1割引
    return total
```

特徴:
- 学習コストが低い
- 短いスクリプト・データ処理向き
- 規模が大きくなると「変数とロジックが散らかる」

**「とりあえず動かす」フェーズに最適**。最初に学ぶスタイル。

## 8-2. オブジェクト指向 (OOP)

**「データ」と「そのデータを操作する関数」をひとまとめにする** 考え方。
そのまとめたものが **オブジェクト (instance)**、その設計図が **クラス (class)** です。

```python
class User:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def greet(self):
        return f"Hi, I'm {self.name}"

u = User("Hikaru", 25)
u.greet()  # → "Hi, I'm Hikaru"
```

### OOP の三本柱

| 概念 | 説明 | 効果 |
|------|------|------|
| **カプセル化** | データを外から触らせず、関数経由で扱う | 内部実装を変更しても影響範囲が小さい |
| **継承** | 既存クラスを拡張して新クラスを作る | 共通処理を再利用 |
| **ポリモーフィズム** | 同じインターフェースで違う実装を呼ぶ | コードを差し替えやすくする |

### 継承の罠

OOP 初学者がやりがちなのが「**継承を多用しすぎる**」こと。深い継承ツリーは後で誰も読めなくなります。
現代の OOP の格言: **「継承より合成 (Composition over Inheritance)」** — 親子関係より、必要な部品をオブジェクトとして持つほうが柔軟。

## 8-3. 関数型 (Functional)

**「データを変更せず、新しいデータを返す」** スタイル。

```python
# 手続き型: リストを直接書き換える
nums = [1, 2, 3, 4, 5]
result = []
for n in nums:
    if n % 2 == 0:
        result.append(n * n)

# 関数型: map と filter で新リストを作る
result = list(map(lambda n: n * n, filter(lambda n: n % 2 == 0, nums)))
# または内包表記で
result = [n * n for n in nums if n % 2 == 0]
```

### 関数型の鍵となる概念

| 概念 | 意味 |
|------|------|
| **イミュータブル** | データを変更しない |
| **純粋関数** | 同じ入力に同じ出力。副作用なし |
| **高階関数** | 関数を引数や戻り値にできる関数 (map, filter, reduce) |
| **再帰** | ループの代わり |

### なぜ関数型が現代で再評価されているのか

- **並行処理に強い**: 状態を変えないので競合が起きにくい
- **テストしやすい**: 純粋関数は引数だけで結果が決まる
- **バグが減る**: 「いつ何が変わったか」を追わなくて済む

純粋な関数型言語 (Haskell, OCaml, F#) は少数ですが、**現代の主要言語はほぼすべて関数型の機能を取り込んでいます** (Python の `map`/`filter`、JavaScript の `map`/`filter`/`reduce`、Kotlin、Swift...)。

## 8-4. 宣言的 vs 命令的

パラダイムを語るときに出てくるもう一つの軸。

- **命令的 (imperative)**: **どうやるか** を一手ずつ書く
- **宣言的 (declarative)**: **何が欲しいか** だけ書く

```sql
-- SQL は宣言的
SELECT name FROM users WHERE age > 20;
```

「年齢が20より大きい人の名前を出して」と要求するだけ。**どうやって探すかはDBに任せる**。

```jsx
// React (JSX) も宣言的
<button>{count}</button>
```

「ボタンを表示して中身は count」と書くだけ。レンダリングの順序や差分計算は React が担当。

宣言的に書けるところは宣言的に書く、というのが現代の流れです。

## 8-5. パラダイムの組み合わせが現代の現場

「Python は OOP、Haskell は関数型」のような単純分類は古いです。**現代の言語は複数パラダイムをサポート (マルチパラダイム)** していて、書き手が場面ごとに使い分けます。

実務での組み合わせ例:

- **アプリの構造**: OOP のクラス (User, Order, Repository...)
- **データ変換**: 関数型の map/filter/reduce
- **イベント処理**: 宣言的な書き方 (React の hooks など)
- **小さなスクリプト**: 手続き型

## 8-6. SOLID 原則 — OOP の指針

OOP を健全に書くための5原則。覚えておくと設計時の判断基準になります。

| 原則 | 一言で |
|------|--------|
| **S**ingle Responsibility | 1クラス1責務 |
| **O**pen/Closed | 拡張に開き、変更に閉じよ |
| **L**iskov Substitution | 子は親の代わりになれ |
| **I**nterface Segregation | 細かいインターフェースを |
| **D**ependency Inversion | 具象でなく抽象に依存 |

詳細はこの章では立ち入りませんが、**「読んで違和感のないコードはだいたいSOLID寄り」** という感覚は持っておくとよいです。

## 8-7. 関数型の頻出 3 関数 — map / filter / reduce

実務でほぼ毎日使うので、ここで体に入れておきましょう。

### map — 各要素を変換する

```python
nums = [1, 2, 3]
doubled = list(map(lambda x: x * 2, nums))  # [2, 4, 6]
# 内包表記なら
doubled = [x * 2 for x in nums]
```

### filter — 条件に合う要素だけ残す

```python
nums = [1, 2, 3, 4, 5]
evens = list(filter(lambda x: x % 2 == 0, nums))  # [2, 4]
# 内包表記なら
evens = [x for x in nums if x % 2 == 0]
```

### reduce — 全体を1つに集約する

```python
from functools import reduce
nums = [1, 2, 3, 4]
total = reduce(lambda acc, x: acc + x, nums, 0)  # 10
```

map と filter で「変換 + 絞り込み」、reduce で「集計」。これだけで多くのデータ処理は表現できます。

## 8-8. パラダイム選びの実用ガイド

| 状況 | 向いているパラダイム |
|------|------------------|
| 短いスクリプト | 手続き型 |
| 大規模アプリの構造 | OOP (+ 一部関数型) |
| データ変換、ETL | 関数型 |
| UI 構築 (React, SwiftUI) | 宣言的 |
| クエリ (DB, GraphQL) | 宣言的 |
| 並行・並列処理 | 関数型 + イミュータブル |

## まとめ

- **パラダイム** = プログラミングの書き方の流派
- **手続き型**: 上から順に書く。素直
- **OOP**: データと関数をオブジェクトにまとめる。**継承より合成**
- **関数型**: 状態を変えず、新しい値を返す。並行に強い
- **宣言的 vs 命令的**: 「何が欲しいか」vs「どうやるか」
- 現代の言語は **マルチパラダイム**。場面ごとに使い分けるのが標準
- **SOLID** は OOP 設計の指針
- **map / filter / reduce** は関数型の三種の神器、毎日使う
