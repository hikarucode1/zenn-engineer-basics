## この章のゴール

2026年現在、エンジニアリングの景色は急速に変わっています。
**ChatGPT、Claude、GitHub Copilot、Cursor、Claude Code** — LLM (大規模言語モデル) を使わずにコードを書くエンジニアは少数派になりました。

しかし「**AI が全部やってくれる**」かというと、全くそうではありません。
むしろ **基礎をしっかり持つエンジニアの価値は上がっています**。AIが書くコードの良し悪しを判断できるのは、結局のところ人間だけだからです。

最終章では、LLM 時代に新人がどう振る舞うか・何を学び続けるか・何を任せて何を任せないか、を扱います。

## 24-1. LLM とは

**LLM (Large Language Model)** = 巨大なテキストデータで訓練された、次の単語を予測するモデル。

主要モデル (2026年時点):
- **Claude** (Anthropic) — 推論・コーディング・長文に強い
- **GPT-4 / GPT-5** (OpenAI) — 汎用、エコシステム広い
- **Gemini** (Google) — マルチモーダル、Google サービス連携
- **Llama** (Meta) — オープンウェイト

ベースは **「次に来る単語の確率分布」を計算しているだけ**、というのが基本原理。これを理解せずに使うと、後述の「ハルシネーション」に振り回されます。

## 24-2. LLM の得意・不得意

### 得意

- **言語化された作業**: コードレビュー、ドキュメント、リファクタ案、命名
- **ボイラープレート**: テンプレ的なコード生成
- **既知パターンへの変換**: ある言語 → 別言語、SQL → ORM
- **要約・抽出**: 長文や巨大ログの中から重要部分を取り出す
- **発想の壁打ち**: アーキテクチャ案、設計の選択肢列挙

### 不得意

- **真の論理的厳密性**: 数式・複雑な制約をミスする
- **最新の情報**: 学習時点以降を知らない (RAG 等で補う)
- **正確な記憶**: 「自信満々で嘘をつく」 = ハルシネーション
- **数百行を超える複雑な状態管理**: コンテキストが長くなるほど精度低下
- **自社固有のドメイン知識**: 当然知らない
- **完全に新しい組み合わせ問題の創造**

## 24-3. ハルシネーション (Hallucination)

LLM が **「もっともらしいが事実と違う」内容を堂々と出力する** 現象。

例:
- 存在しないライブラリのメソッドを使ったコード
- ありそうな URL や論文タイトル
- 古いAPI のシグネチャを最新だと言う

対策:
- **必ず実行・検証する** (動かす、ドキュメント確認する)
- 重要な事実は **複数ソースで裏取り**
- 「**自信のある語り方**」に騙されない

「**LLM は自分の知らないことを知らない**」と覚えてください。

## 24-4. プロンプトの基本

LLM への指示の出し方。Garbage In, Garbage Out。

### 良いプロンプトの要素

1. **役割**: 「あなたは経験豊富な Web セキュリティエンジニアです」
2. **タスク**: 「次のコードの脆弱性を指摘してください」
3. **コンテキスト**: 関連コード、制約、目的
4. **形式**: 「箇条書きで」「JSON で」「3つに絞って」
5. **例 (Few-shot)**: 期待するアウトプットの例を1〜2個

### 反パターン

- 「いい感じにして」 → 何の手がかりもない
- 「全部やって」 → 範囲が広すぎて精度ダウン
- 巨大ファイル丸投げ → 重要部分が薄まる

## 24-5. RAG — モデルの知識を外部から補う

**Retrieval-Augmented Generation** = 「**質問に応じて関連ドキュメントを検索して、それを LLM に渡す**」仕組み。

```
ユーザー質問
   ↓
ベクトル検索 (関連ドキュメントを取得)
   ↓
[質問 + 関連ドキュメント] → LLM
   ↓
回答
```

利点:
- 最新情報・社内情報を反映できる
- ハルシネーションが減る
- 出典を提示できる

社内QA、カスタマーサポート、コードベースの理解など、現代のAIアプリの主要パターン。

## 24-6. エージェント — LLM がツールを使う

「LLM が **ツール (関数、API、コマンド) を呼びながら多段で動く**」のがエージェント。

例: コードエージェント
- ファイルを読む
- テストを走らせる
- エラーを見て修正
- もう一度テスト
- 通ったらコミット

Claude Code、Cursor の Agent モードなどがこれ。**自然言語で指示を出して、コードベースを編集してもらう** のが現代の開発スタイルになりつつあります。

## 24-7. LLM に何を任せて、何を任せないか

新人ほど線引きが重要。

| 任せていい | 任せない方がいい |
|----------|----------------|
| 初稿の生成 | 設計判断 |
| ボイラープレート | セキュリティ要件 |
| テストケースのたたき | 本番リリース判断 |
| ドキュメント | コードレビューの最終判断 |
| ログ・エラーの解読補助 | お金が絡む処理 |
| 命名候補出し | 仕様の解釈 |

**「LLM が書いたコードを読まずに本番に出すのは、見知らぬインターンの PR を一発マージする」のと同じです**。

## 24-8. AI 時代の新人エンジニアに必要な能力

LLM が出てきたから不要になった能力 = ない。
LLM が出てきたから **特に重要になった能力**:

1. **読む力**: AI が出したコードを正確に理解する
2. **検証する力**: 動くか・正しいか・安全かを確かめる
3. **設計判断**: 何を作るか・どう作るかは人間の仕事
4. **領域知識 (ドメイン)**: 業務理解は AI には引き出しにない
5. **質問を立てる力**: 何を聞くかは AI には決められない
6. **倫理・責任**: 何を作るべきでないか

**「コードを書ける」から「コードを書かせて、評価できる」** への重心移動が起きています。

## 24-9. AI 倫理と責任

LLM を組み込むときに考えるべきこと。

- **個人情報を渡していいか** (社外SaaSへの送信)
- **著作権**: 学習データ由来の問題、生成物の利用範囲
- **偏見・差別**: モデルが持つ統計的偏り
- **誤情報の責任**: 「AI が言った」は免責にならない
- **透明性**: ユーザーに「AI が生成した」を伝えるか

EU AI Act など、各国で規制も整いつつあります。**「AI に任せた」では責任を逃れられない**、という前提を持ってください。

## 24-10. 学び続けるエンジニアへ

ここまでで24章、現代の Web エンジニアリングの基礎が一通り見えたはずです。
最後に、これからどう学んでいくかのアドバイスを。

### 1. 浅く広く → 深く狭く

本書のような俯瞰本で全体像を掴んだら、**興味のある領域を1つ深掘り** する。
広く知っているだけでは差別化されない。「**X についてはあいつに聞こう**」と呼ばれる領域を作る。

### 2. 作る・公開する

学んだことは **動くもの・公開されたもの** にして初めて定着します。

- 個人プロジェクト
- OSS にコントリビュート
- Zenn / Qiita に記事を書く
- 登壇する

「**作って公開した数 = エンジニアとしての筋肉**」と考えてください。

### 3. 一次情報を読む

ChatGPT に聞くのも便利ですが、**公式ドキュメント・RFC・論文・ソースコード** は信頼度が桁違いです。
特に「最新仕様」「正確な仕様」が必要な場面では一次情報をたどる癖をつける。

### 4. コードを読む量を増やす

書くことの数倍、読むほうが学びになります。

- 自分が使っているライブラリのソース
- 有名 OSS の構造
- 同僚の PR

**「読めるコード量 ≈ 書けるコード量」** という経験則があります。

### 5. 健康と継続

エンジニアリングは長距離走。

- 睡眠を削らない
- 運動する
- 趣味を持つ
- 焦らない

「**1日10ページの読書を10年続けたほうが、3ヶ月で100時間詰め込むより遠くに行ける**」 — そう信じて続けてください。

## 最後に

エンジニアリングは、技術スタックが毎年入れ替わる慌ただしい世界です。
でも本書で扱った **基礎の8割は、20年前も20年後も変わりません**。

- コンピュータは CPU と メモリと I/O で動く
- データは 0 と 1 だ
- HTTP はリクエストとレスポンスだ
- DB は ACID だ
- セキュリティは多層防御だ
- テストは設計の道具だ
- 自分のコードは自分で運用する

新しい言語、新しいフレームワーク、新しいAIモデルが出てきても、これらの土台の上に乗ります。
**土台を持つあなたは、変化の速い時代でも迷子にならない**。

新人時代のあなたが、5年後、10年後にもう一度この本を開いて「**ああ、ここに書いてあった**」と笑える日が来ることを願って、筆を置きます。

長い道のりをお疲れさまでした。
ここからの旅は、コードと一緒に。
