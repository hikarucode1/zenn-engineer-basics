## この章のゴール

Git は **エンジニアの呼吸** です。
コードの歴史を残し、複数人で同じコードベースを編集し、いつでも過去に戻れる — これがないと現代の開発は1日も回りません。

この章では Git の概念モデル・基本操作・ブランチ戦略・コンフリクト解消・GitHub PR フローを扱います。

## 20-1. バージョン管理とは

「**いつ、誰が、何を、なぜ変えたか**」を記録する仕組み。

これがないと:
- 「先週はちゃんと動いていたのに...」を再現できない
- 複数人で同じファイルを編集できない
- 過去のリリースに戻せない
- 誰が何を変えたかわからない

Git 以前に SVN, CVS などもありましたが、**現代のデファクトは完全に Git**。

## 20-2. Git のメンタルモデル

「ファイルを保存する」ではなく、**「変更のスナップショットを積み重ねる」** のが Git の本質。

```
[コミット1] ← [コミット2] ← [コミット3] (HEAD)
   ↑              ↑               ↑
プロジェクト      プロジェクト        プロジェクト
の状態1          の状態2            の状態3
```

各コミットには:
- ハッシュ (一意のID、`a3f2c1...`)
- 変更内容
- 作者
- 日時
- メッセージ

## 20-3. 3つのエリア

新人が最も詰まる Git の概念。

```
[作業ディレクトリ] ─add→ [ステージング] ─commit→ [リポジトリ]
   (まだ未追跡)        (次にコミットする内容)      (履歴に追加)
```

| エリア | 何が入る |
|-------|---------|
| **作業ディレクトリ (Working)** | 今編集中のファイル |
| **ステージング (Index)** | コミット対象として選んだ変更 |
| **リポジトリ (HEAD)** | 確定した履歴 |

`git add` でステージに上げ、`git commit` でリポジトリに確定。
「**ステージング == コミットの下書き**」というイメージが分かると Git 全体が見えてきます。

## 20-4. 必須コマンド

| コマンド | 何をする |
|---------|---------|
| `git init` | リポジトリ初期化 |
| `git clone <url>` | 既存リポジトリをコピー |
| `git status` | 現在の状態を確認 |
| `git add <file>` | ステージング |
| `git commit -m "msg"` | コミット作成 |
| `git log` | 履歴表示 |
| `git diff` | 変更内容を確認 |
| `git branch` | ブランチ一覧 |
| `git checkout <branch>` / `git switch <branch>` | ブランチ切替 |
| `git checkout -b <branch>` / `git switch -c <branch>` | 新規ブランチ作成 |
| `git merge <branch>` | ブランチをマージ |
| `git pull` | リモートから取得+マージ |
| `git push` | リモートに送る |

「迷ったら `git status`」が新人時代の合言葉。今どこにいて何が起きているかを確認。

## 20-5. ブランチ — 並行作業の基本単位

ブランチは **コミット履歴に付けた可動の "しおり"**。
複数の作業を独立して進めるために使う。

```
main:     A → B → C
                  ↘
feature:          D → E
```

- `main`: 安定版
- `feature/login`: 新機能の開発線

別ブランチで作業すれば、`main` を壊さずに済む。完成したら `merge` で本流に戻す。

## 20-6. マージとリベース

ブランチを統合する2つの方法。

### マージ (merge)

「**もう1つの履歴があった**」事実を残す。

```
main:     A → B → C ──→ M
                  ↘    ↗
feature:           D → E
```

`M` は **マージコミット**。両方の親を持つ。

### リベース (rebase)

「**自分の作業を、相手の最新の続きとして書き直す**」。履歴が直線になる。

```
main:     A → B → C
                  ↘
feature:           D' → E'  (BとCの上にやり直し)
```

| | merge | rebase |
|--|-------|--------|
| 履歴 | 残る (枝分かれが見える) | 直線になる |
| 安全性 | ◎ | △ (公開済みは要注意) |
| 推奨 | チーム合流 | 自分のローカル整理 |

**ルール: 既に push したコミットを rebase しない**。チームの履歴が壊れる。

## 20-7. コンフリクト解消

別ブランチで同じ行を変更すると **コンフリクト** が発生。

```
<<<<<<< HEAD
これは main の変更
=======
これは feature の変更
>>>>>>> feature
```

対応:
1. ファイルを開き、`<<<` ~ `>>>` の中身を見て **正しい内容に書き直す**
2. マーカーを消す
3. `git add <file>`
4. `git commit` (マージ中なら自動メッセージ)

新人は怖がりがちですが、慣れれば普通の作業。落ち着いて1ファイルずつ。

## 20-8. リモートとローカル

| 場所 | 例 |
|------|----|
| ローカル | あなたのMac/PCの中 |
| リモート | GitHub / GitLab / Bitbucket のサーバー |

リモートに上げるのが `git push`、リモートから取るのが `git fetch` / `git pull`。

`git pull` = `git fetch` + `git merge` (または rebase)。
中身を意識するために初心者でも `git fetch` → `git status` → `git merge` と分けて打つ練習をおすすめします。

## 20-9. GitHub と Pull Request フロー

業界標準ワークフロー。

1. リポジトリを **clone** または **fork**
2. ブランチを切る (`feature/xxx`)
3. 作業 → コミット → push
4. GitHub で **Pull Request (PR)** を作成
5. 他人がレビュー、コメント・修正
6. 通れば **マージ**
7. ブランチ削除

PR は「コードレビューの場」「議論の記録」「マージの前提チェック」を兼ねる。
**「動いた」ではなく「読みやすい・理由が分かる」コード** を出すのが PR を出す側の作法。

### コミットメッセージのお作法

「**何を変えたか + なぜ変えたか**」を簡潔に。
Conventional Commits という命名規則が広く使われています。

```
feat: ユーザー登録時にメール認証を追加
fix: ログイン後のリダイレクトループを修正
docs: README にセットアップ手順を追記
refactor: User クラスを Auth と Profile に分割
test: 注文集計のテストを追加
chore: 依存ライブラリを更新
```

`feat / fix / docs / refactor / test / chore / style / perf / build / ci` あたりがよく使われる prefix。

## 20-10. .gitignore — Git に入れたくないもの

リポジトリに入れるべきでないものをここに書く。

```
# 環境変数
.env
.env.local

# 依存関係
node_modules/
__pycache__/
*.pyc

# OS が作る不要ファイル
.DS_Store
Thumbs.db

# ビルド成果物
dist/
build/

# IDE 固有
.vscode/
.idea/
```

**秘密情報を間違って commit する** のは新人あるある。push 前に `git diff --cached` でチェックを習慣化。

## 20-11. やらかしたときの復旧

### 直前のコミットをやり直す

```bash
git commit --amend -m "新しいメッセージ"
```

ただし push 済みなら他人に影響するので注意。

### コミットを打ち消す

```bash
git revert <commit>
```

「打ち消すコミット」を新規作成。push 済みでも安全。

### 作業ディレクトリを最後のコミット状態に戻す

```bash
git checkout -- <file>  # 単一ファイル
git reset --hard HEAD   # 全部 (破壊的!)
```

:::message alert
`reset --hard` は **コミットしていない変更を完全に消します**。実行前に必ず `git status` で本当に消していいか確認。
:::

### コミットを過去にさかのぼって直す (歴史改変)

```bash
git rebase -i HEAD~3   # 直近3コミットを編集
```

ローカルでの整理用。push 済みは触らない。

### 一時退避

```bash
git stash      # 今の変更を退避
git stash pop  # 戻す
```

「途中まで作業したけど別ブランチに飛びたい」ときに便利。

## 20-12. ブランチ戦略

チームでよく使うパターン。

### Git Flow

main / develop / feature / release / hotfix ブランチを使い分け、リリース管理が厳密。大規模・ウォーターフォール寄り向き。

### GitHub Flow

シンプル。**main + 機能ブランチ + PR + マージ + デプロイ** の繰り返し。スタートアップやWebサービスの定番。

### Trunk-Based Development

短命ブランチで頻繁に main に統合。CI/CD が強い前提。Google などが採用。

新人がいる現場では **GitHub Flow** が学びやすく、トラブルも少ないのでおすすめ。

## まとめ

- Git は **変更スナップショットの積み重ね** で歴史を残す
- 3つのエリア: **作業ディレクトリ / ステージング / リポジトリ**
- ブランチは並行作業の単位、main を壊さないで済む
- **マージは履歴を残す、リベースは直線にする**。push済みはrebase禁止
- コンフリクトは怖くない、`<<< === >>>` を直して `add → commit`
- **GitHub Flow**: ブランチ → PR → レビュー → マージ
- コミットメッセージは Conventional Commits で意味が分かる形に
- `.gitignore` で機密 / 依存 / 一時ファイルを除外
- やらかしたら `amend / revert / reset / stash` を使い分ける
- ブランチ戦略は **GitHub Flow** から始めるのが現実的
