## この章のゴール

データ構造が「箱」なら、アルゴリズムは「**箱に対する手順**」です。

「100万件のリストから特定のユーザーを見つける」「商品を価格順に並べる」 — どんなアプリでも避けて通れないこれらを、現場で通用するレベルで理解するのが本章のゴール。

数学的な厳密さより、**「なぜそれが速いのか」「いつ使うか」** を肌で掴むことを優先します。

## 7-1. 計算量 (Big-O) — 「だいたいどれくらい遅くなるか」

データ量が増えたとき、処理時間がどう増えるかを表すのが **計算量** で、`O(...)` の記号で書きます。

| 表記 | 名前 | 例 | n=1000のとき |
|------|------|----|-------------|
| O(1) | 定数時間 | 配列の i 番目 | 1 |
| O(log n) | 対数時間 | 二分探索 | 10 |
| O(n) | 線形時間 | 配列を全部見る | 1,000 |
| O(n log n) | 線形対数 | マージソート | 10,000 |
| O(n²) | 二乗時間 | バブルソート | 1,000,000 |
| O(2ⁿ) | 指数時間 | 全組み合わせ | 天文学的 |

ポイントは、**n が大きくなったとき何が支配的か** を見ること。`n² + 100n` は O(n²) です (定数や低次項は無視)。

:::message
「学生時代に出てきた話」と侮らないこと。現場で「データが増えたら急に遅くなった」と言われる8割はこの計算量の見落としです。
:::

## 7-2. 線形探索 (Linear Search) — まず単純に

配列の中から特定の値を探す、最も単純な方法。

```python
def linear_search(arr, target):
    for i, v in enumerate(arr):
        if v == target:
            return i
    return -1
```

計算量: **O(n)**。100万件あれば最悪100万回比較。
シンプルで間違えにくい。データが小さいうちはこれで十分。

## 7-3. 二分探索 (Binary Search) — 「半分に絞る」の威力

**事前にソートされた配列** に対してだけ使える手法。真ん中を見て「左右どっちにあるか」で半分に絞っていきます。

```python
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
```

計算量: **O(log n)**。
100万件 → 約20回で答えが出ます。線形探索 (100万回) と比べて 5万倍 速い。

これが第13章のDBインデックスの根本原理です。

## 7-4. ソート — 並べ替えの基本

「並べ替える」だけでも何種類もアルゴリズムがあります。代表選手3つ。

### バブルソート (O(n²)) — 学習用

隣同士を比べて大きいほうを右へ送る、を繰り返す。

```python
def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(n - i - 1):
            if arr[j] > arr[j+1]:
                arr[j], arr[j+1] = arr[j+1], arr[j]
```

シンプルだが遅い。**「ソートとは何か」を理解する以外には使わない**。

### マージソート (O(n log n))

配列を半分に分け、それぞれをソートして、最後に **マージ (合体)** する。

```
[5,2,8,3,1,9,4,7]
        ↓ 分割
[5,2,8,3] [1,9,4,7]
        ↓ さらに分割
[5,2][8,3][1,9][4,7]
        ↓ マージしながら戻る
[2,5][3,8][1,9][4,7]
[2,3,5,8][1,4,7,9]
[1,2,3,4,5,7,8,9]
```

**分割統治法** という考え方の代表例。安定して速く、現場でもよく使われます。

### クイックソート (平均 O(n log n))

「ピボット」と呼ぶ基準値を1つ選び、それより小さいグループと大きいグループに分けて再帰的にソート。

平均的にはマージソートと同等の速さで、メモリ使用量が少ない。多くの言語の標準ソートはこれかその変種を使っています。

### 言語の標準ソートを使え

業務で自分でソートを書くことはほぼありません。`sorted(arr)` を呼べば言語が最適化された実装をしてくれます。**「中身は O(n log n) で書かれている」** と知っておけば十分です。

## 7-5. 再帰 (Recursion)

関数が **自分自身を呼ぶ** ことを再帰と呼びます。木やグラフを扱うアルゴリズムでは必須の考え方。

```python
def factorial(n):
    if n <= 1:
        return 1
    return n * factorial(n - 1)
```

再帰には必ず **ベースケース (止める条件)** が必要。ないと無限に呼び続けて **スタックオーバーフロー** で落ちます (第3章のスタックを思い出してください)。

## 7-6. 探索アルゴリズム — グラフを歩く

第6章のグラフを「歩く」方法は大きく2つ。

### 幅優先探索 (BFS) — キューを使う

スタート地点から近い順に探索。**最短経路** を求めたいときに使う。

```
       [A]
       /|\
      B C D
     /|   |\
    E F   G H
```

BFS の訪問順: A → B → C → D → E → F → G → H

実装は **キュー** で「次に訪れるノード」を管理。

### 深さ優先探索 (DFS) — スタックを使う

一本の道を行けるところまで進み、行き止まりで戻る。

訪問順: A → B → E → F → C → D → G → H

実装は **再帰** か **スタック**。迷路の解、組み合わせ列挙などで使う。

## 7-7. 動的計画法 (DP) — 一度解いたものは再利用する

同じ計算を何度もしないように、結果を **メモ (キャッシュ)** しながら進める手法。

例: フィボナッチ数列。素直に再帰で書くと O(2ⁿ) かかります。

```python
# 素直に書いた版 (遅い)
def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)

# メモ化版 (O(n))
cache = {}
def fib_dp(n):
    if n <= 1: return n
    if n in cache: return cache[n]
    cache[n] = fib_dp(n-1) + fib_dp(n-2)
    return cache[n]
```

「**重複した計算をやめる**」だけで指数時間が線形時間になります。LeetCode などで頻出のテーマ。

## 7-8. 貪欲法 (Greedy) — 目の前のベストを選ぶ

「**毎ステップ、その時点で最良の選択をする**」戦略。全体最適にならないケースもありますが、シンプルで高速。

例: お釣り問題 — 500円玉から順に出していく。

正解になる問題と失敗する問題がはっきり分かれるので、**「貪欲で解けるか?」を見極める力** が必要です。

## 7-9. ハッシュを使った高速化テクニック

「集合に入っているか」「重複があるか」を高速判定するとき、**ハッシュマップ/集合に入れて O(1) で引く** のが定番テクニック。

```python
# 二重ループで O(n²) (遅い)
def has_pair_with_sum(arr, target):
    for i in range(len(arr)):
        for j in range(i+1, len(arr)):
            if arr[i] + arr[j] == target:
                return True
    return False

# ハッシュで O(n) (速い)
def has_pair_with_sum_fast(arr, target):
    seen = set()
    for v in arr:
        if (target - v) in seen:
            return True
        seen.add(v)
    return False
```

「O(n²) → O(n) に落とすコツの8割はハッシュ」と覚えておくと、実務で効きます。

## 7-10. アルゴリズム選定の実務センス

新人がつまずきがちなのが、**「LeetCode で求められる完璧さ」と「実務で求められるバランス」のギャップ** です。

実務では:

- データ量が小さい → 単純な O(n²) でも全然OK
- 1日1回のバッチ処理 → 多少遅くてもOK
- リアルタイム応答 (API) → ミリ秒単位で気にする
- 大規模データ (1億件) → 計算量が命

**「全部最適化しろ」ではなく「ボトルネックを見つけて、そこを最適化しろ」** が現場の作法です。

## まとめ

- **計算量 Big-O** は「n が増えたときどう遅くなるか」を表す共通言語
- **線形探索 O(n)** より **二分探索 O(log n)** が桁違いに速い (要ソート済み)
- **マージソート / クイックソート** は O(n log n)。実装は標準ライブラリに任せる
- **再帰** は強力だがベースケース必須。スタックオーバーフローに注意
- **BFS (キュー)** で最短経路、**DFS (スタック / 再帰)** で組み合わせ
- **動的計画法 (DP)** = 重複計算をやめる。指数時間を線形時間に
- **ハッシュ集合** で O(n²) を O(n) に落とすのが現場の頻出テクニック
- 実務では「**全部最適化」より「ボトルネックを潰す**」
