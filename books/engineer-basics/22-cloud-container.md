## この章のゴール

「サーバーを買う・置く」時代から「**必要なときに借りる**」時代に変わったのがクラウド。
「アプリと環境を丸ごとパッケージ化」して動かすのがコンテナ。

この章では AWS / GCP / Azure の基本サービス、Docker、Kubernetes、サーバーレス、マネージドサービスの考え方を扱います。
膨大な領域なので、**「最低限知っておくべき地図」** を提供します。

## 22-1. クラウドが解決すること

オンプレ (自前サーバー) の悩み:
- サーバーを物理で買う → 数十万〜
- データセンターに置く → 場所代・電気代
- 容量が足りなくなる → 増設、また数ヶ月
- ピーク時に合わせて大きく買うと、暇な時間は遊ぶ
- 1台壊れると詰む

クラウドの解:
- API でサーバーを **数分で立ち上げ・破棄**
- 使った分だけ課金 (時間単位や秒単位)
- 必要に応じて自動で増減 (オートスケーリング)
- 複数地域に冗長化

## 22-2. 三大クラウド (Big 3)

| 提供者 | 強み |
|-------|------|
| **AWS** (Amazon) | サービス数最多、業界デファクト |
| **GCP** (Google) | データ分析・AI、Kubernetes 発祥 |
| **Azure** (Microsoft) | エンタープライズ、Office との統合 |

新人がまず触るなら **AWS** が情報量・採用市場でも有利。ただし概念はどれも似ているので、ひとつ深く知れば他も理解できます。

## 22-3. クラウドの3階層モデル

| 階層 | 提供されるもの | 例 |
|------|--------------|----|
| **IaaS** (Infrastructure) | サーバー、ストレージ | EC2, S3, GCE |
| **PaaS** (Platform) | 実行環境 | Heroku, AWS Elastic Beanstalk |
| **SaaS** (Software) | 完成品 | Gmail, Slack |

下に行くほど自由・上に行くほど楽。
最近は **マネージドサービス** が増え、IaaS と PaaS の境界は曖昧になっています。

## 22-4. AWS の最頻出サービス

新人がまず覚えるべきもの。

| サービス | 役割 |
|---------|------|
| **EC2** | 仮想サーバー |
| **S3** | オブジェクトストレージ (ファイル、画像) |
| **RDS** | マネージド RDB (PostgreSQL/MySQL) |
| **DynamoDB** | マネージド NoSQL KVS |
| **Lambda** | サーバーレス関数 |
| **CloudFront** | CDN |
| **Route 53** | DNS |
| **VPC** | 仮想ネットワーク |
| **IAM** | アクセス権限管理 |
| **CloudWatch** | ログ・メトリクス |
| **SQS** | メッセージキュー |
| **SNS** | 通知 |
| **API Gateway** | API のフロント |
| **ECS / EKS** | コンテナ実行環境 |

数百あるサービスのうち、これくらいを知っていれば普通の Web サービスは作れます。

## 22-5. リージョンとアベイラビリティゾーン

- **リージョン (Region)**: 地理的な場所 (東京、バージニア、シンガポール…)
- **アベイラビリティゾーン (AZ)**: リージョン内の独立した物理データセンター (通常2〜3個)

冗長化の基本は **「複数 AZ に配置」**。1 つの AZ が停電・障害でも他で動き続ける。
グローバルサービスは **複数リージョン** にも展開し、ユーザーに近い場所から応答する。

## 22-6. Docker — コンテナの代名詞

**コンテナ = アプリと依存環境を1つにパックした実行単位**。

```
[ アプリ (例: Python) ]
[ ライブラリ・依存パッケージ ]
[ OS の最小ファイルセット ]
        ↓
[ Docker エンジン ]  ← 1台のサーバー上で複数コンテナを並行実行
[ ホストOS (Linux) ]
```

仮想マシン (VM) と違って **OS まるごとは複製しない**。だからずっと軽く・速く起動できる。

### Dockerfile

コンテナのレシピ。

```dockerfile
FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .

CMD ["python", "app.py"]
```

これをビルド → イメージ → 配布 → どこでも同じように動く。

### Docker のメリット

- **「自分のマシンでは動いた」問題が消える**
- ローカル / ステージ / 本番で同じ環境
- マイクロサービス、CI、開発環境の標準
- 起動が秒単位

新人エンジニアがチームに入る初日に `docker compose up` で全環境が立ち上がるのが現代の理想。

## 22-7. Docker Compose

複数コンテナをまとめて立ち上げる仕組み。

```yaml
services:
  web:
    build: .
    ports: ["8000:8000"]
    depends_on: [db, redis]

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret

  redis:
    image: redis:7
```

`docker compose up` でアプリ + DB + Redis が一発で起動。
ローカル開発で必須レベルの道具。

## 22-8. Kubernetes (K8s) — コンテナのオーケストレーション

「100個のコンテナを、何台のサーバーにどう配って、死んだら再起動して、トラフィックに応じて増減させる」 を自動でやってくれるプラットフォーム。

主な概念:
- **Pod**: コンテナを1〜数個まとめた最小単位
- **Deployment**: Pod を何個動かすかの管理
- **Service**: Pod へのアクセスを安定化 (ロードバランサ)
- **Ingress**: 外部からのルーティング
- **ConfigMap / Secret**: 設定・シークレット管理

学習コスト高。**小〜中規模なら ECS や App Runner で十分**。大規模・マイクロサービスで真価を発揮します。

## 22-9. サーバーレス

「サーバーを意識せず、関数 (or コンテナ) を置くだけで動く」モデル。

### FaaS (Function as a Service)

- **AWS Lambda**
- **Google Cloud Functions**
- **Cloudflare Workers**

```python
def handler(event, context):
    name = event["name"]
    return {"statusCode": 200, "body": f"Hello {name}"}
```

イベント (HTTPリクエスト、ファイルアップロード、スケジュール…) を起点に関数が走る。
**使った分だけ料金。アイドル時間は0円**。

向く用途:
- 軽い API
- バッチ処理
- イベント駆動 (アップロード時にサムネ生成)

向かない用途:
- 起動時間に敏感な低レイテンシ (コールドスタート)
- 大量の長時間接続 (WebSocket は工夫要)

### コンテナ系サーバーレス

- **AWS Fargate** / **Google Cloud Run**: コンテナを置くだけ。サーバー管理不要

「**Cloud Run / Fargate に Docker イメージを置く**」が現代の Web アプリの定番パターン。

## 22-10. マネージドサービスを使う原則

「**自分で運用する vs マネージドを使う**」の判断軸。

| 自前運用 (例: EC2 に MySQL を立てる) | マネージド (例: RDS) |
|---|---|
| ◯ 安い | △ 高い |
| × バックアップ・冗長化・パッチを自分で | ◯ 全部やってくれる |
| × 障害対応が深夜の自分 | ◯ AWS が対応 |
| ◎ 自由度高 | △ 制約あり |

「**コア機能じゃない部分はマネージドに丸投げ**」 — エンジニアリソースを自社の本業に集中させる、というのが現代の経済的合理性。

## 22-11. クラウドコストの感覚

クラウドは便利ですが、**油断すると簡単に高額請求** が来ます。

新人がやりがちな事故:
- 巨大インスタンスを起動して止め忘れる
- S3 に放置 → 数年で数百GB
- ログを INFO レベルで大量出力 → CloudWatch 料金
- データ転送料 (リージョン跨ぎ、外向き)

防衛:
- **コストアラート** を設定 (Budget)
- **未使用リソースの定期チェック**
- **使い終わったら必ず破棄** (terraform destroy など)

## 22-12. クラウドの責任共有モデル

セキュリティ責任を「**クラウド事業者 / 利用者**」で分担する考え方。

| 担当 | 例 |
|------|----|
| クラウド事業者 (AWS など) | 物理セキュリティ、ハイパーバイザ、データセンター |
| 利用者 (あなた) | OS パッチ、ネットワーク設定、アプリ、データ、IAM |

「クラウドだから安全」は半分嘘。**利用者が設定をミスると公開状態** になり得ます。
S3 を public read 設定で公開 → 個人情報流出、というニュースは毎年あります。

## まとめ

- クラウド = 「**必要な時に借りる**」、使った分課金、自動で増減
- **AWS / GCP / Azure** が三大、まずは AWS から触ると情報多い
- **IaaS / PaaS / SaaS** の3階層、最近はマネージドが主流
- **複数 AZ** に配置して冗長化
- **Docker** はアプリ + 依存をパック。「自分のマシンでは動いた」が消える
- **Compose** で複数コンテナを一発起動、開発環境の標準
- **Kubernetes** はオーケストレーション、本格規模で
- **サーバーレス**: Lambda / Cloud Run / Fargate、アイドル0円
- **マネージドサービス** はコア以外を任せて時間を買う
- コストは油断すると爆発、アラート・破棄習慣を
- **責任共有モデル**: 設定は利用者責任、公開ミスに注意
