## この章のゴール

ここから第3部「ネットワークと Web」。
スマホで `google.com` と打って Enter を押すと、なぜ世界の裏側にあるサーバーから 1秒以内に検索結果が返ってくるのか — その仕組みを支えているのが **ネットワーク** です。

この章では Web の土台 — **IP・TCP・UDP・DNS・ポート** を扱います。

## 9-1. ネットワークは「層」で考える

ネットワークの世界は驚くほど多くのプロトコルが絡みます。整理するために **OSI 7階層モデル** と、その実用簡略版である **TCP/IP 4階層モデル** が使われます。

| TCP/IP 階層 | 役割 | 代表プロトコル |
|------------|------|-------------|
| アプリケーション層 | アプリ同士の会話 | HTTP, HTTPS, FTP, SMTP, DNS |
| トランスポート層 | データを届ける方法 | TCP, UDP |
| インターネット層 | アドレス指定とルーティング | IP, ICMP |
| ネットワークインターフェース層 | 物理的な伝送 | Ethernet, Wi-Fi |

「層」がある理由は **責任分離** です。上の層は下の層の詳細を知らなくてよい。**HTTP を書くアプリは、それが Wi-Fi を通っているか光ファイバーを通っているか意識する必要がない** — これが層の威力です。

## 9-2. IP アドレス — ネットワーク上の住所

世界中の機器に振られる **住所** が IP アドレスです。

### IPv4

`192.168.1.10` のように4つの数字 (0〜255) で表す。約43億通り。
すでに枯渇しており、家庭内では **NAT** で同じグローバルIPを複数台で共有しています。

### IPv6

`2001:0db8:85a3:0000:0000:8a2e:0370:7334` のように16進数8グループ。事実上無限。徐々に普及中。

### プライベート IP と グローバル IP

| 種類 | 例 | 用途 |
|------|----|----|
| グローバル IP | 8.8.8.8 | インターネット上で一意 |
| プライベート IP | 192.168.x.x, 10.x.x.x | 家庭・社内LANで使う |
| ループバック | 127.0.0.1 | 自分自身 |

自宅でルーターを開くと `192.168.1.1` が出てくる、`localhost = 127.0.0.1` は自分のマシンを指す、と覚えてください。

## 9-3. ポート — 同じIPの中のドア

1台のサーバーで複数のアプリ (Web、メール、SSH...) が動いています。これを区別するのが **ポート番号** (0〜65535)。

| ポート | サービス |
|-------|---------|
| 22 | SSH |
| 25 | SMTP (メール送信) |
| 53 | DNS |
| 80 | HTTP |
| 443 | HTTPS |
| 3306 | MySQL |
| 5432 | PostgreSQL |
| 6379 | Redis |
| 8080 | HTTP (代替) |

IP アドレスがマンションの住所、ポートが部屋番号、と例えるとイメージしやすいです。

## 9-4. TCP — 信頼性のあるデータ転送

**TCP (Transmission Control Protocol)** は、**確実にデータを届ける** ためのプロトコル。

特徴:
- **コネクション型**: 通信を始める前に握手 (3-way handshake)
- **再送**: パケットが届かなければ送り直す
- **順序保証**: 届いたデータを正しい順に並べる
- **フロー制御**: 受信側が処理できる速度に合わせる

### 3-way handshake

```
クライアント        サーバー
    | --- SYN ---> |
    | <- SYN/ACK - |
    | --- ACK ---> |
    | 接続確立        |
```

Web (HTTP/HTTPS)、メール、SSH、DB通信 — 普段使うものはほぼ TCP です。

## 9-5. UDP — 速さ優先のデータ転送

**UDP (User Datagram Protocol)** は、**信頼性を捨てて速さを取る** プロトコル。

特徴:
- **コネクションレス**: いきなり送る
- **再送なし**: 届かなくても気にしない
- **順序保証なし**

「届かなくてもいいから速く!」という用途に向いています。

| 用途 | なぜ UDP か |
|------|------------|
| ライブ動画/音声 | 1フレーム落ちても全体は流したい |
| DNS | 一発で問い合わせて返ってくる |
| ゲーム | 古い位置情報は捨てて最新を |
| IoT / LoRaWAN | 帯域が貴重で TCP のオーバヘッドが重い |

:::message
卒研などで LoRaWAN を扱う場合、ペイロードは UDP 系で運ばれます。TCP のような握手を省けるのが低帯域・低消費電力デバイスにはありがたい。
:::

## 9-6. DNS — 「名前 → IP」の電話帳

`google.com` のような **ドメイン名** を、`142.250.207.78` のような IP アドレスに変換するのが **DNS (Domain Name System)** です。

IP アドレスは覚えづらいので、人間用の名前と機械用の番号を分離した、ともいえます。

### 名前解決の流れ

1. ブラウザがOSに `google.com` を問い合わせ
2. OSは DNS リゾルバ (ISPなどが提供) に問い合わせ
3. リゾルバが **ルートDNS → TLD DNS (.com) → 権威DNS (google.com)** を順にたどる
4. IP が返ってくる
5. 結果をキャッシュ (次回は速い)

### DNS レコードの種類

| レコード | 意味 |
|---------|------|
| A | ドメイン → IPv4 |
| AAAA | ドメイン → IPv6 |
| CNAME | 別名 (エイリアス) |
| MX | メール用サーバー |
| TXT | 任意のテキスト (認証情報の検証など) |
| NS | 権威DNSサーバー |

「ドメインを買って Web サイトを公開する」とき、最初に触るのがDNS設定 (A レコードに自分のサーバーのIPを書く) です。

## 9-7. クライアントとサーバー

ネットワーク通信の基本構造は **クライアント・サーバーモデル**。

```
[クライアント] --- リクエスト ---> [サーバー]
[クライアント] <-- レスポンス ---  [サーバー]
```

- **クライアント**: 要求する側 (ブラウザ、スマホアプリ)
- **サーバー**: 応答する側 (Webサーバー、DB、API)

例外的に **P2P (Peer-to-Peer)** は中心サーバーなしで端末同士が直接通信します (BitTorrent、一部のチャット)。

## 9-8. NAT・ファイアウォール・プロキシ

### NAT (Network Address Translation)

家庭内の複数端末が1つのグローバルIPを共有できる仕組み。ルーターが内側 (`192.168.x.x`) と外側 (グローバルIP) のアドレスを変換しています。

### ファイアウォール

通信を許可/拒否する関所。
- 特定のポート (例: 80, 443) だけ通す
- 特定の IP からの接続だけ通す/拒否する

セキュリティの第一防衛線で、第17章でも再登場します。

### プロキシ

クライアントの代理でサーバーへアクセスする中継役。
- 社内ネットの出口に立ってフィルタリング
- キャッシュで高速化
- API のリバースプロキシ (Nginx など)

## 9-9. CDN — 世界中にコンテンツを配る

**CDN (Content Delivery Network)** は、画像・動画などの静的コンテンツを **世界中のサーバーに配置** しておき、ユーザーから最も近い場所から配信する仕組み。

```
ユーザー (日本) → CDNエッジ (東京) ← オリジン (米国)
```

CloudFront, Cloudflare, Fastly などが代表。Web パフォーマンスのためにほぼ必須の存在になっています。

## 9-10. ネットワークが遅い・繋がらないときの切り分け

新人がよくぶつかる「アクセスできない」問題の切り分け順:

1. **物理層 (Wi-Fi/有線)**: 接続マークは出ているか
2. **IP**: `ping 8.8.8.8` で Google DNS に届くか
3. **DNS**: `ping google.com` でドメインが解決できるか
4. **ポート/ファイアウォール**: `curl https://...` で実際にアクセスできるか
5. **アプリ層**: HTTP のレスポンスは何を返しているか (第10章)

下から順に切り分けるのが鉄則。

## まとめ

- ネットワークは **層 (アプリ / トランスポート / インターネット / 物理)** で考える
- **IP** が住所、**ポート** が部屋番号
- **TCP**: 信頼性重視、再送あり、Webやメールはこれ
- **UDP**: 速さ重視、再送なし、ライブ動画やDNSはこれ
- **DNS** が「名前→IP」を変換。レコード種類は A, CNAME, MX...
- **クライアント・サーバー** が基本構造、**P2P** は例外
- **NAT / ファイアウォール / プロキシ** で通信を制御
- **CDN** で世界中に配信を高速化
- 切り分けは **物理 → IP → DNS → ポート → アプリ層** の順

次の章では、Web の主役 — **HTTP** の仕組みに踏み込みます。
